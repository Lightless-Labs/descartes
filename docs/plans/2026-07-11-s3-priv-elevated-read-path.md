## 0. Grounding — what's real today

Confirmed by direct repo inspection (not assumed from the Layer B plan's prose):

- **There are TWO distinct degrade shapes, and the elevated path must target the RIGHT one (corrected in refine — the draft conflated them).** Verified directly against the live code:
  - **Graceful cross-UID / unresolved-pid degrade** (the case S3-priv's elevated path is *for*): `resolveByPortLinux` L698–711 builds `resolvedStatus:'partial'` with the owning `uid` still present, which `computeProvenanceEnvelopeFields` (L456–461) maps to **`status:'partial'`, `confidence:0.4`, `review_hint:'missing_permission'`**. This is pinned by `test/provenance.test.js:326` (`computeProvenanceEnvelopeFields('partial','unknown') → {status:'partial', confidence:0.4, reviewHint:'missing_permission'}`). The owning UID is a *free confident fact* the unprivileged path already has.
  - **Exceptional fail-closed** (thrown exception mid-collection): only `timedEnvelope`'s catch path produces **`status:'unable'`, `confidence:0`, `review_hint:'missing_permission'`**, exercised by `test/provenance.test.js:334–343` with a deliberately-throwing hostile input.
  - **The invariant for every S3-priv elevated-path FAILURE is therefore: fall back to and RETURN the already-computed unprivileged `partial`/`0.4`/`missing_permission` result for that target — never collapse to the more-degraded `unable`/`0` shape, which would discard the owning-UID fact the unprivileged path already resolved for free.** `unable`/`0` remains reserved for genuine thrown-exception cases (`timedEnvelope`'s existing contract, untouched). A draft that asserted "every failure degrades to `unable`/`confidence:0`" would have made Slice 6's CI baseline assertion fail against the real code on first run — this refine corrects it throughout (Slice 2 TDD, Slice 6 step 6, post-flight item 3, §4).
- **Node plumbing seam is well-defined.** `tools/descartes-cli/src/tools/provenance.js`: `resolveByPortLinux` (~L658–712) has the exact cross-UID degrade branch — `if (currentUid !== undefined && primary.uid === currentUid) { ...own-uid fd-walk... }`, else falls through unconditionally to a hardcoded partial-degrade result via `finalizeProvenanceResult`. `emptyPrivilege()` (~L467–469) returns a hardcoded `{mechanism:'unprivileged', elevated_available:false, elevated_used:false}` consumed by three separate builders (`finalizeProvenanceResult`, `buildNotFoundResult`, `buildInvalidTargetResult`) — all three need to accept a *computed* privilege value, from one shared computation, not three divergent inline edits. `resolveProvenance(params={})` (~L756) has no second `options`/`paths` argument today; `pi-harness.js` (~L199) calls it with a single arg. `resolveByPortMac` (~L643–656) has **no** cross-UID gap today (unprivileged `lsof` already resolves pid regardless of owning UID) — its analogous need, if any, is `codesign`/`spctl` on other-UID executables, which is explicitly out of scope pending the §6(f) macOS spike.
- **DI precedent is exactly one shape, everywhere.** `provenance-identity.js` (S5, shipped): `async function defaultStatExecutablePath(x) { return fs.stat(x); }` then `const statExecutablePath = options.statExecutablePath ?? defaultStatExecutablePath;`, tested via a plain `fakeCollectorOptions()` object literal (`test/provenance-identity.test.js:49-57`). Zero `jest.mock`/`sinon`/`node:test mock.method` usage anywhere in `tools/descartes-cli/test/*.js`. S3-priv's mock-helper injection must copy this pattern verbatim — no new mocking mechanism.

### Slice 2 — Elevated-path plumbing with a mock helper (the core Node logic)

**Goal:** Implement the full two-condition opt-in gate, probe, fixed-argv invocation, echo-back scope verification, trust-boundary check, and degrade-to-unprivileged logic — entirely against an **injectable mock helper**, exactly mirroring `provenance-identity.js`'s `options.<fn> ?? default<Fn>` DI pattern. No real helper binary exists yet; this slice defines the helper's I/O *contract* as an executable specification (the mock IS that contract).

**locallyTestable:** true (100% mock-driven, no privilege, runs on macOS or Linux CI unchanged).

**Artifacts:**
- `tools/descartes-cli/src/provenance-elevated.js` (new) — the elevated-path module:
  - `defaultProbeElevatedHelper(helperPath)` — `execFile(helperPath, ['--probe'], {timeout, maxBuffer, env: MINIMAL})` where `MINIMAL` is an explicit minimal environment (`{PATH: '/usr/bin:/bin'}`, nothing inherited from the daemon's ambient `process.env`) — defense-in-depth against `LD_PRELOAD`/`LD_LIBRARY_PATH`-class injection into a capability-bearing child, not relying solely on the kernel's `AT_SECURE`/secure-execution mode. Catches `ENOENT`/`EACCES`/`EPERM`/timeout, returns `{available:boolean}`. Cached per process lifetime (probe result memoized, not re-probed per call). **The cache is only a hint/optimization: the authoritative gate is invoke-time OS enforcement (the kernel's capability check at `exec`/`ptrace` time), not the cached probe. A grant revoked mid-run (group membership removed, capability stripped) leaves a stale `available:true`, but the next real invoke still fails-closed with EPERM/ENOENT and degrades — the cache can never cause a fabricated success.** Consider building on the existing `runFixedExecFile(command, args, options)` (provenance.js L52–70, already the bounded-timeout/maxBuffer array-argv exec used for `docker`/`podman inspect`) rather than a parallel `execFile` wrapper, or document an explicit rationale for divergence — avoid two slightly-diverging "shell out safely" implementations in the same file.
  - `defaultInvokeElevatedHelper(helperPath, argv)` — `execFile(helperPath, argv, {timeout, maxBuffer: BOUNDED, env: MINIMAL})` (same minimal-env hardening as probe). `argv` is always one of the two literal bounded forms: `['--resolve-port', String(port)]` or `['--resolve-pid', String(pid)]` — **never** a template/interpolated string, asserted by a dedicated test that spies the call and checks `Array.isArray` + exact literal shape.
  - `verifyTrustBoundary({configPath, configDir, helperPath}, statFn)` — rejects if the config file, its parent dir, or the helper binary is group/world-writable (`mode & 0o022`) or not owned by the expected principal; **AND walks the helper binary's full ancestor-directory chain (parent, grandparent, … up to root), rejecting if any ancestor directory is group/world-writable or not root-owned** — on POSIX, write access to a *containing* directory lets a local user unlink/replace the file inside it regardless of the file's own bits, the classic dir-hijack local-privesc vector, and it is worse here because the named `root_helper` fallback runs the replaced binary AS ROOT. This mirrors the SSH `authorized_keys`/host-key directory-chain check discipline. Returns `{trusted:boolean, reason}`. Fail-closed: any stat error (ENOENT on helper, an unreadable ancestor, etc.) also returns `trusted:false`.
  - `parseAndVerifyHelperResponse(rawStdout, requestedTarget)` — bounded-length parse (reject if `rawStdout.length` exceeds a fixed cap before even attempting `JSON.parse`), schema-validated shape (closed set of expected fields, no extra-field passthrough), and **echo-back scope verification**: the parsed response must include and exactly match the requested `{pid}` or `{port}` — any mismatch (different pid/port than requested) is treated identically to a parse failure, never merged.
  - `resolveElevated({target, config, paths, unprivilegedResult}, options = {})` — the orchestrator: two-condition AND gate (`config.elevated.enabled === true` AND probe reports `available:true`), trust-boundary check, invoke, parse+verify. **On verified success it merges the resolved pid/executable/etc. onto the target's result. On ANY non-success exit path (helper absent, config disabled, probe fail, trust-boundary reject, timeout, malformed/oversized/scope-mismatch response) it returns the already-computed unprivileged result unchanged — i.e. the graceful `partial`/`confidence:0.4`/`review_hint:'missing_permission'` shape with the owning UID still present — never a distinct, more-degraded `unable`/`0` surface (that shape is reserved for `timedEnvelope`'s thrown-exception path and is left untouched).** This is why the caller passes `unprivilegedResult` in: the elevated attempt *upgrades* an already-good partial result or falls back to it; it never discards the free UID fact. `mechanism:'auto'` never selects `root_helper` as a probe target (only `cap_sys_ptrace`/`helper_xpc`-equivalent probes are attempted under `auto`; `root_helper` is only probed when explicitly named in config).
  - **Residual TOCTOU (accepted risk, documented):** there is an inherent check-then-use window between `verifyTrustBoundary`'s `stat()` calls and the subsequent `execFile()` of the helper — a directory/file swap in between could bypass the check. Marginal risk is low because the precondition (transient write access to a root-owned protected path or its ancestors) already implies serious compromise; it is accepted here and noted so a reviewer needn't infer it. Where feasible, the Rust helper (Slice 3/4) should prefer `O_NOFOLLOW`/`openat`-based acquisition over check-then-open to shrink the window from the privileged side.
- `tools/descartes-cli/src/tools/provenance.js` (edit) — wire `resolveElevated` into `resolveByPortLinux`'s cross-UID branch so that it **wraps** the existing partial-degrade: first compute the current unprivileged partial result (L698–711, the `partial`/owning-UID fact), then hand that result to `resolveElevated({..., unprivilegedResult})` as the fallback it returns on any non-success. This is a pure refactor first (extract the cross-UID handling into its own small function, behavior-preservation-tested) before layering the elevated *upgrade* on top — keeping the diff reviewable per the ground map's recommendation, and structurally guaranteeing the free UID fact is never discarded on a degrade.
- `tools/descartes-cli/src/tools/provenance.js` (edit) — widen `resolveProvenance(params, options = {})` to accept an optional second arg carrying `paths` and DI overrides (`probeElevatedHelper`, `invokeElevatedHelper`, `loadProvenanceConfig`, `statFn`), threaded down through `resolveByTarget → resolveByPort → resolveByPortLinux`. Additive only — the existing single-arg call shape must continue to produce byte-identical output.
- `tools/descartes-cli/src/pi-harness.js` (edit) — thread `paths` into the `resolveProvenance` call at the tool executor (L199), matching the existing two-positional-arg convention (`sampleDimensionEvidence(params, paths)`).
- `tools/descartes-cli/test/provenance-elevated.test.js` (new) — `fakeElevatedOptions({...})` helper, plain object literal, no mocking library.

**TDD test list (from Layer B plan §3, items 1–11, all executed here with mocks):**
1. Helper absent (`ENOENT`) → `elevated_available:false`, falls through to unprivileged, no throw.
2. Config disabled but helper present → still falls through (config wins; opt-in is a two-key AND, both keys tested independently: helper-present/config-disabled AND helper-absent/config-enabled).
3. Config enabled + helper present + probe succeeds → `elevated_used:true`; on a verified cross-UID resolution the result **upgrades** from `partial`/`0.4` to `ok`/`1` with pid/executable now populated, plus `privilege.mechanism` set. (The unprivileged baseline for the same target is `partial`/`0.4`/`missing_permission` with the owning UID present — the elevated success upgrades that, it does not replace an `unable`/`0`.)
4. Fixed-argv assertion: spy on the invoke call, assert argv is a literal bounded array, never a template string.
5. Probe timeout → degrades by **returning the unprivileged `partial`/`0.4`/`missing_permission` result unchanged**, does not hang the collector (fake that never resolves, bounded by an injected short timeout in test).
6. **Regression:** with `elevated.enabled:false` (the shipped default), the full pre-existing `provenance.test.js` suite output is byte-identical.
7. *(Deferred to Slice 5 — real-host diagnostic, not mockable):* `ptrace_scope` probe.
8. Malformed JSON, or JSON exceeding the bounded length cap → degrades by returning the unprivileged `partial`/`confidence:0.4`/`review_hint:'missing_permission'` result unchanged (owning UID preserved), never a partial-parse merge and never a collapse to `unable`/`0`.
9. Well-formed response for a **different** pid/port than requested (simulated race/bug/compromise) → echo-back mismatch detected, mismatched fact never merged, degrades by returning the unprivileged `partial`/`0.4`/`missing_permission` result unchanged.
10. `helper_path` pointing at a group/world-writable file, **or at a file inside a group/world-writable / non-root-owned ANCESTOR directory**, or a config file in a group/world-writable directory → rejected by `verifyTrustBoundary`, degrades by returning the unprivileged `partial` result, does not execute the path, logs a warning (fixture stat-mode values across the full path chain, no real elevated file needed). Includes a dedicated case where the leaf binary is correctly `0750 root:root` but its **parent dir** is `0777` — must still reject (the dir-hijack vector).
11. `mechanism:"auto"` with only `root_helper` "available" (per a fake probe) does **not** auto-select it — stays unprivileged (returns the unprivileged `partial`) unless `root_helper` is named explicitly in config.
12. **Minimal-env assertion:** spy on both the probe and invoke `execFile` calls and assert they are passed an explicit minimal `env` (e.g. `{PATH:'/usr/bin:/bin'}`), NOT the daemon's ambient `process.env` — a positive test that ambient environment does not leak into the capability-bearing child.
13. **Signature widening regression:** every pre-existing call site of `resolveProvenance` (single-arg) and `pi-harness.js`'s executor continue to behave identically; a dedicated test exercises the executor's actual `params → resolveProvenance` call (closing the ground map's flagged gap that `pi-harness.test.js`'s tool-name-set-equality check alone would not catch a broken `paths` threading).

**Acceptance criteria:**
- Default install (no config, no helper): behavior is byte-identical to pre-S3-priv S3 — verified by running the full pre-existing suite unmodified against the new code path with default config.
- No code path in this slice ever shells to a privilege-escalation mechanism or invokes an OS admin-prompt API — verified structurally by a **standing CI lint** (see post-flight), not a one-time diff grep. The lint's term list must be broad enough for the eventual macOS work too: `sudo`, `setcap`, `pkexec`, `doas`, `osascript.*administrator`, `SMJobBless`, `SMAppService`, `AuthorizationExecuteWithPrivileges` — scoped to `provenance.js`/`daemon.js`/`provenance-elevated.js` with an explicit allowlist for Slice 5's docs and Slice 6's disposable CI script.
- `emptyPrivilege()`'s three call sites are unified behind one `computePrivilege` function (Slice 1) — no divergent per-site privilege population.
- Untrusted helper stdout is never merged without passing both the bounded-parse and the echo-back scope check.

**Safety notes:** This is the highest-value, lowest-risk slice — all the security-critical *logic* (opt-in gate, trust boundary, echo-back verification, degrade-not-fabricate) is implemented and fully tested here, before any real privilege exists anywhere in the system. A reviewer can audit the entire decision tree against a mock, independent of Rust/CI work.

### Slice 3 — Rust `root_helper` crate: skeleton, fixed-argv contract, JSON stdout

**Goal:** Stand up the first Rust crate in the repo, matching exactly the contract Slice 2's mock already encodes: `--resolve-port <n>` / `--resolve-pid <n>`, JSON stdout, nothing else. No hardening yet (that's Slice 4) — this slice is "does it build, does it speak the contract, does it live in a Bazel-friendly location."

**locallyTestable:** true, but only on a machine with a Rust toolchain (not necessarily macOS-first here, since the target is Linux `/proc`; can be developed/unit-tested for argv-parsing and JSON-shape logic on any host, but the actual `/proc` resolution logic is Linux-only and should be `#[cfg(target_os = "linux")]`-gated with a stub/error on other platforms).

**Artifacts:**
- `crates/descartes-root-helper/Cargo.toml`, `crates/descartes-root-helper/src/main.rs` (new) — location chosen to match `docs/HANDOFF.md`'s documented (not-yet-built) target shape (`crates/descartes-core`, `crates/descartes-collector` siblings), keeping the crate graph legible for an eventual Bazel migration per `AGENTS.md`'s "if this repo starts with Cargo, keep it Bazel-friendly: no hidden build steps, checked-in manifests, reproducible tests, clear crate graph" guidance.
- Root-level `Cargo.toml` workspace manifest (new) — even a one-crate workspace, so the crate graph is explicit and future crates (`descartes-core` etc.) join it without restructuring.
- Fixed-argv parser: rejects anything other than exactly `--resolve-port <digits>` or `--resolve-pid <digits>` or `--probe`; no flag combinators, no `--help`-driven dynamic behavior beyond a literal usage string, no shell/env-based configuration (no config file parsing, per the settled design).
- JSON stdout writer: emits a closed-shape object echoing the requested target plus resolved fields (`{requested:{port|pid}, resolved:{pid, uid, executable_path, ...}}`) — the exact echo-back shape Slice 2's `parseAndVerifyHelperResponse` expects. This is the one place the "contract" is authored on both sides; a shared fixture (checked into both the Node test file and a Rust integration test as literal JSON) keeps them from drifting.
- `crates/descartes-root-helper/tests/` — Rust-side tests for argv rejection (garbage argv → nonzero exit, no partial output), JSON shape (golden fixture match), and `/proc` resolution against a known test pid (the test's own process).

**TDD / acceptance criteria:**
1. `cargo test` passes for argv-rejection and JSON-shape cases without requiring elevated privilege (the crate's own process can read its own `/proc` entry unprivileged — this is what makes the crate testable without root in a normal dev/CI unit-test pass; the *cross-UID* case is what needs the real capability, deferred to Slice 6's CI job).
2. `--resolve-pid <own-pid>` returns a JSON blob matching the shared fixture shape exactly.
3. Any argv other than the two literal forms exits nonzero with no stdout (never partial/malformed output on bad input).
4. No `unsafe` blocks beyond what's strictly required for the `/proc`-reading and (later, Slice 4) syscall-restriction code — flagged explicitly as a reviewable subset of the diff.
5. Crate builds and tests green on the `ci-linux-arm64-rust-bazel` image once toolchain presence is confirmed (Slice 6 pre-flight) — not assumed here.

**Safety notes:** This slice deliberately does **not** grant any privilege to the built binary — it is a plain unprivileged executable at this point, testable exactly like any other unprivileged Rust binary. No `setcap` anywhere in this slice.

### Slice 4 — Hardened build/test pipeline (seccomp-bpf, no-new-privs, cap-drop) — first-class acceptance items

**Goal:** Implement and, critically, **kernel-level test** (not just code-review-assert) the three required hardening properties from Layer B §6(b): capability-drop-after-open, `PR_SET_NO_NEW_PRIVS`, and a seccomp-bpf syscall allowlist that structurally prevents any `PTRACE_PEEKDATA`/`PTRACE_POKEDATA`/`PTRACE_CONT` call from ever being reachable. These are acceptance items in the **crate's own pipeline**, explicitly not the Node `npm test` suite (per the settled design, restated here because it is easy to silently drop).

**locallyTestable:** true on Linux with a Rust toolchain; **not** meaningfully testable on macOS (seccomp-bpf and Linux capabilities are Linux-only) — this slice's real tests run in Slice 6's Linux CI job, but the hardening *code* and non-privileged unit tests around it (e.g., "does the allowlist reject an unlisted syscall in a controlled harness") can be authored and partially exercised in any Linux dev environment first.

**Artifacts:**
- Capability-drop: after opening the needed `/proc/<pid>/...` paths for the current call, drop all capabilities except the bare minimum still needed to finish writing stdout (the capability is held only for a scoping window, not the process lifetime). Prefer `O_NOFOLLOW`/`openat`-based acquisition of those `/proc` paths over check-then-open patterns, shrinking the residual TOCTOU window flagged in Slice 2 from the privileged side.
- `PR_SET_NO_NEW_PRIVS` set unconditionally at process start, before any argv-driven logic runs.
- Seccomp-bpf allowlist — **derived empirically, NOT hand-enumerated.** The draft's illustrative list (`open`/`openat`/`readlink`/`read`/`close`/`exit`) is a *first-pass placeholder only*: as written it omits `write` (the whole point of the binary is to emit JSON to stdout) and every baseline Rust-runtime syscall a trivial Rust binary needs for its allocator/panic machinery/HashMap seeding (`mmap`, `mprotect`, `munmap`, `brk`, `futex`, `rt_sigaction`, `rt_sigprocmask`, `sigaltstack`, `clock_gettime`, `getrandom`, `sched_getaffinity`, …). Locked in as literally drafted, the hardened binary would SIGSYS-kill *itself* during ordinary operation, not just under attack — and the resulting build-breakage pressure would invite exactly the "loosen the allowlist just to ship" unreviewed change the design elsewhere forbids. **Acceptance requirement:** derive the real allowlist by `strace -f` (or `seccomp` audit mode) against the *actual compiled release binary* exercising BOTH the `--probe` path AND both `--resolve-port`/`--resolve-pid` paths; check the derived allowlist into the crate as data alongside a test that fails loudly if the binary's observed syscall set drifts outside it. The invariant that matters is the *denials*, tested positively below: `ptrace` and all network syscalls (`socket`/`connect`/`sendto`/…) are **absent from the allowlist**, so any attempt kills the process via SIGSYS regardless of code path.
- `crates/descartes-root-helper/tests/hardening.rs` — runs the built binary under the actual seccomp filter and asserts: (a) a normal `--resolve-pid <own-pid>` call succeeds end-to-end (including its `write` to stdout — proving the empirically-derived allowlist is complete, not just restrictive); (b) **[no-ptrace]** a `#[cfg(test)]`-only harness that calls `ptrace(PTRACE_PEEKDATA, ...)` directly is killed by the kernel (SIGSYS), not merely returns an error; (c) **[no-network]** a symmetric `#[cfg(test)]`-only harness that attempts `socket()`/`connect()` is *also* killed by the kernel (SIGSYS) — so the design's "no-network" hard requirement is verified with the identical kernel-enforced rigor as "no-ptrace", not merely implied by allowlist omission. Both (b) and (c) are the concrete "enforced at the kernel level, not just by code review" assertions the settled design requires.
- `getcap`-visibility check (Linux) — a test/script step that runs `getcap` on the built binary after a test-time `setcap` and asserts the expected capability string, confirming the binary is `setcap`-compatible (does not itself grant the capability in CI at build time — that's Slice 6).

**TDD / acceptance criteria (first-class, per the settled design — restated so they cannot be silently skipped):**
1. Capability-drop-after-open is unit-tested (a controlled harness observes the process's effective capability set before and after the drop point).
2. `PR_SET_NO_NEW_PRIVS` is asserted set via `prctl(PR_GET_NO_NEW_PRIVS)` read-back in a test.
3. **Seccomp allowlist completeness:** a normal `--probe` and both `--resolve-*` calls succeed *under the live filter* (proves the empirically-derived allowlist, including `write` and the Rust-runtime baseline syscalls, is complete — the binary does not kill itself). A drift test fails loudly if the release binary's observed syscall set falls outside the checked-in derived allowlist.
4. **Seccomp `no-ptrace` rejection is a kernel-enforced test** (process killed by SIGSYS on a deliberate `ptrace` attempt), not a code-review assertion or a mocked syscall table.
5. **Seccomp `no-network` rejection is a kernel-enforced test** (process killed by SIGSYS on a deliberate `socket()`/`connect()` attempt) — the "no-network" requirement proven with the same rigor as "no-ptrace".
6. PR diff for this slice is auditable at "a few hundred lines at most" per the settled design's own stated size expectation — if it grows past that, the design should be revisited, not the size limit silently abandoned.
7. All of the above run in the crate's own `cargo test` (or an equivalent dedicated script), explicitly listed in this slice's CI step (Slice 6) as separate from `npm test`.

**Safety notes:** This is the slice where "CAP_SYS_PTRACE's true blast radius is full ptrace, not a scoped procfs read" (§6(b)) gets its actual mitigation. A memory-safety bug or crafted input in this binary, *without* this slice's hardening, has full-ptrace blast radius; *with* it, the kernel itself refuses the dangerous syscalls regardless of what the Rust code tries to do. Treat any weakening of the seccomp allowlist (e.g., adding a syscall "just in case") as a security-review-required change, not a routine PR.

### Slice 5 — Documented install-time privilege + OS-level audience-scoping

**Goal:** Since no automated Linux installer/packaging pipeline exists in this repo today (confirmed, zero hits), and the settled design explicitly treats a post-install-as-root script as a **future** option, this slice is **documentation + a verification script**, not new application code. It records the manual operator grant and gives the operator (and CI) a way to check it's correctly scoped, plus wires the real (non-mock) `ptrace_scope` diagnostic deferred from Slice 2.

**locallyTestable:** true for the diagnostic script (read-only, runs unprivileged, works on any Linux host including CI); the actual grant itself is inherently not locally-unit-testable (it requires root and is a one-time manual act) — that reality is *by design*, not a gap this slice tries to paper over.

**Artifacts:**
- `docs/operator/linux-elevated-provenance-setup.md` (new) or a section appended to `README`/`HANDOFF` — the exact manual command sequence:
  1. `groupadd --system descartes-provenance` (dedicated group, not a general admin group, per §6(d)).
  2. Install the built+hardened helper binary to a fixed path, `chown root:descartes-provenance <path>`, `chmod 0750 <path>` (not world-executable).
  3. `setcap cap_sys_ptrace=ep <path>` (primary mechanism).
  4. Add the account the Descartes daemon runs as to the `descartes-provenance` group.
  5. Set `configDir/provenance.json`'s `elevated.enabled:true` and `mechanism` (explicit `auto` for capability-scoped, or `root_helper` named explicitly) — the second, independent AND-condition.
  6. Named-fallback (`root_helper` via systemd) documented separately: `User=root`, `CapabilityBoundingSet=CAP_SYS_PTRACE` only, `NoNewPrivileges=yes`, `ProtectSystem=strict`, socket-activated, scoped to the exact binary path with no argument substitution — or an equivalently narrow sudoers/polkit rule. Explicitly **not** wired to `mechanism:"auto"`. **The activation socket unit MUST be audience-scoped at the OS level, at least as tight as the primary capability binary's `0750 root:descartes-provenance` — since this mechanism runs the helper as literal root, unscoped it would let any local process reaching the socket trigger a root-run invocation.** Required socket hardening: `SocketMode=0660`, `SocketUser=root`, `SocketGroup=descartes-provenance` (only members of that dedicated group may write to the activation socket) — this is the higher-privilege mechanism's equivalent of the file's `0750` mode and must not be left unspecified. (Open question 7 still governs the *choice* of systemd-vs-sudoers/polkit; this fix governs the audience-scoping of whichever is chosen.)
- `crates/descartes-root-helper/scripts/verify-install.sh` (new, read-only) — checks: helper path exists, owned `root:descartes-provenance`, mode `0750` (not world-executable/world-readable-execute); **every ancestor directory of the helper path is root-owned and not group/world-writable** (dir-hijack guard, the shell mirror of Slice 2's `verifyTrustBoundary` ancestor walk); `getcap` shows exactly `cap_sys_ptrace=ep` and nothing broader; config file/parent dir not group/world-writable; **and, if the named `root_helper`/systemd fallback is configured, the `.socket` unit's `SocketMode`/`SocketUser`/`SocketGroup` match the scoped values above (not world-writable, group `descartes-provenance`).** Exits nonzero with a clear operator-facing message on any mismatch — this script is what Slice 2's `verifyTrustBoundary` logic mirrors in spirit (same checks incl. the ancestor-directory walk, one on the daemon's hot path in JS, one as an operator-facing standalone diagnostic in shell/Rust).
- `tools/descartes-cli/src/provenance-elevated.js` (edit) — wire the real (non-mock) `ptrace_scope` diagnostic: `fs.readFile('/proc/sys/kernel/yama/ptrace_scope')`, logged/surfaced as a diagnostic field, never a hard-fail (matches Layer B §3 TDD item 7 — "a diagnostic, not a hard-fail, since this varies per host").
- `tools/descartes-cli/test/provenance-elevated.test.js` (edit) — add the `ptrace_scope` diagnostic test using a fake `statFn`/`readFile` (still mock-driven; this is a value-surfacing test, not a real-host assertion — the real-host assertion is Slice 6's CI job).

**TDD / acceptance criteria:**
1. `ptrace_scope` diagnostic surfaces the real value when readable, degrades silently (field omitted, not a crash) when unreadable (e.g., non-Linux, or the file doesn't exist on an unusual kernel config).
2. `verify-install.sh` correctly flags each of: missing binary, wrong owner, wrong group, world-executable mode, missing/wrong `getcap` output, group/world-writable config, **a group/world-writable or non-root-owned ancestor directory of the helper path (dir-hijack case — leaf bits correct but parent dir `0777`), and (when the systemd fallback is configured) a `.socket` unit whose `SocketMode`/`SocketGroup` are broader than `0660`/`descartes-provenance`** — one fixture-driven pass per failure mode, run against a locally-created scratch file/dir (no real setcap needed to test the *checking* logic, only real setcap needed to test the *positive* case, which is exercised in Slice 6's CI job, not here).
3. Documentation is reviewed against the exact settled-design wording (dedicated group, not general admin group; `0750` not world-executable; named not auto for `root_helper`) — a docs-accuracy check, not a code test, but listed here because it is a merge-blocking acceptance item per the doors-and-corners discipline below.

**Safety notes:** This slice is explicitly **not** an automated installer. It is intentionally manual because building an automated root-requiring installer/packaging pipeline is out of scope (it would be its own prerequisite slice, per the ground map's own finding) and because the settled design treats that as a named *future* option, not a v1 requirement. The graceful-degrade-when-absent behavior (Slice 2) is what makes shipping without an installer safe: the feature is fully inert until an operator does this manual sequence *and* flips the config flag.

### Slice 6 — Linux-CI validation: build, grant, cross-UID assertion

**Goal:** A new, dedicated Buildkite step that proves the elevated path actually resolves cross-UID ownership that the unprivileged path reports as `missing_permission` — the one thing no amount of mock-driven Node testing or crate-level Rust testing can prove, because it requires the real capability grant and a real second UID.

**locallyTestable:** false in the "run on a laptop" sense (requires root in a disposable Linux environment) — but **is** exercisable in a disposable Linux container/VM during development (e.g., locally in a Linux Docker container with `--cap-add=SYS_ADMIN` equivalent testing, or directly on the tart-ci guest via a manual Buildkite trigger) before being wired as a standing CI gate. Flagged explicitly: this slice's "local" testability is bounded by needing *some* Linux root, not necessarily the CI guest specifically.

**Artifacts:**
- A **new** Buildkite step, e.g. `label: ":linux: Linux ARM64 (elevated provenance)"`, `key: linux-arm64-elevated`, same `queue: ci-linux-arm64` / same `image: ci-linux-arm64-rust-bazel`, joining the **same** `concurrency_group: "big-cabbage/tart-ci"` (per the ground map's explicit risk note — do not oversubscribe the host-wide cap of 2 with an uncoordinated new group). Gated `if: build.tag == null`, same as the existing `linux-arm64` job (runs on every push, not tag-only, since this validates ordinary code changes to the elevated path, not a release artifact).
- Step logic (new, this repo's first-ever privileged/multi-UID CI step):
  1. **Pre-flight probe** (see doors-and-corners below): `sudo -n true` — if this fails, the step must fail loudly with an actionable message ("CI user lacks passwordless sudo in this guest — coordinate with the tart-ci image owner"), not silently skip. This turns an unverified assumption into an explicit, visible CI signal rather than a false green.
  2. `cargo build --release` for `crates/descartes-root-helper` (Rust toolchain-availability is itself asserted here, not assumed — `cargo --version` checked first with a clear failure message if absent).
  3. Run Slice 4's hardening test suite (`cargo test`) — non-privileged parts.
  4. As root (via the pre-flight-verified sudo, and only after the pre-flight sysadmin-binary probe has confirmed `setcap`/`getcap`/`groupadd` exist on the guest): create the `descartes-provenance` group, `chown`/`chmod` the built binary per Slice 5's documented sequence, `setcap cap_sys_ptrace=ep` it.
  5. Create a second, different-UID test user (`useradd` or `setpriv --reuid` — both confirmed present by the pre-flight probe, not assumed; these ship in shadow-utils/util-linux which are frequently absent from minimal/slim images), spawn a fixture listener process under that UID bound to a test port.
  6. Run the **unprivileged** Node path (default config) against that cross-UID port — assert the existing, already-shipped S3 baseline shape: **`status:'partial'`, `confidence:0.4`, `review_hint:'missing_permission'`, with the owning `uid` present and `pid` undefined** (this is exactly what `resolveByPortLinux` L698–711 → `computeProvenanceEnvelopeFields` produce today, pinned by `test/provenance.test.js:326`; asserting `unable`/`0` here would fail against the real code — see §0). This is the "before" baseline in the same job that proves the "after".
  7. Run the **elevated** Node path (config `elevated.enabled:true`, real helper path, real mechanism) against the same port — assert it **upgrades** that baseline: resolves the correct owning uid/pid/executable_path (`status:'ok'`, `confidence:1`), and `privilege.mechanism`/`elevated_used:true` are populated correctly. The delta between step 6 and step 7 (partial→ok, pid appears) is the cross-UID proof.
  8. Run `verify-install.sh` (Slice 5) against the real grant, asserting it reports healthy.
  9. Teardown: kill the fixture listener, delete the test user — all inside the ephemeral per-job VM, which is discarded anyway (belt-and-suspenders, not load-bearing, since the whole clone is deleted post-job).

**TDD / acceptance criteria:**
1. The job fails clearly and specifically if sudo is unavailable, rather than passing trivially by skipping the privileged assertions.
2. The job fails clearly and specifically if the Rust toolchain is unavailable, rather than assuming the image name guarantees it.
3. The cross-UID assertion is the actual gate: unprivileged path on the cross-UID port → `missing_permission`; elevated path on the same port → correct resolution. Both assertions live in the same job so they can't silently drift apart (e.g., someone "fixing" the elevated path by accidentally also fixing the unprivileged path, masking a real regression).
4. `ptrace_scope` value on the real CI guest is logged (diagnostic, per Slice 5) — if it turns out to be `2` or `3` (admin-only/disabled) on this specific image, that is a **finding to record**, not a silent pass/fail; the plan does not assume a specific value.
5. Job runtime and cold-`cargo build` cost are measured on first run and noted (no Rust caching precedent exists in this repo yet) — not blocking, but tracked so it doesn't silently become a CI-slowness surprise.

**Safety notes:** This is the only slice in the whole plan that runs anything as root, and it does so **only** inside an ephemeral, single-purpose, host-capped CI VM that is destroyed after the job — never on a developer's machine, never on a long-lived host. If sudo is unavailable in the guest, this slice's implementation is **blocked**, not silently descoped — see Pre-flight checklist.

## 3. macOS go/no-go spike (explicitly time-boxed, NOT a POC — separate from this plan's Slices 1–6)

Per operator decision, the macOS half of S3-priv is **not scheduled in this plan's slices**. Layer B §6(f) already specifies the checkpoint; restated here only to keep this plan self-contained and to prevent scope creep into macOS code before the spike runs:

- **Form:** a short, read-only investigation using S3's already-shipped unprivileged macOS collector against real fixtures — **not** a working prototype of any macOS helper, XPC service, or `SMAppService` registration. No new macOS code is written during the spike itself.
- **Time-box:** a single bounded working session (per the memory note "spikes stay time-boxed" — a decision aid, not a weeks-long hardened build). If the spike cannot reach a clear answer within that box, the default disposition is "defer, gap unproven" rather than extending the spike.
- **Investigates exactly three cases** (per §6(f)): (a) `codesign -dv`/`spctl` behavior on another UID's executable, (b) a SIP-protected or root-owned `launchd` job's listening port, (c) a container-runtime helper process's port — determining what fraction of realistic "why is this running / who owns this port" queries the existing unprivileged macOS path (which already resolves pid via `lsof` regardless of owning UID) already answers versus what remains genuinely blocked by cross-UID kernel restrictions.
- **Valid outcomes, all equally acceptable, recorded as a written decision appended to Layer B plan §9:**
  1. Proceed with the full `SMAppService`/XPC helper (greenfield in this repo — no `SMJobBless`/`SMAppService`/XPC precedent exists today).
  2. Descope to a `libproc`-direct `root_helper`-equivalent only (no XPC).
  3. **Defer/skip macOS entirely** — "gap too narrow" is a valid, successful outcome, not a failure to reach a decision.
- **If a macOS mechanism is chosen**, it gets its **own** fresh `doors-and-corners` pass at that implementation time (not reuse of this plan's or Layer B's planning-time pass) and its **own** dedicated implementation plan (this plan does not pre-author macOS slices) — matching this plan's own Slices 1–6 sequencing discipline (Node plumbing with a mock helper first, then the native binary, then install-time privilege, then CI validation) once/if it is greenlit.

## 4. Safety section (summary, cross-referenced to slices)

- **Daemon never self-escalates.** Structurally enforced (Slice 2) and structurally verified (post-flight checklist) — the long-running Node process only ever probes and invokes a fixed-argv helper; it never calls `sudo`/`setcap`/an admin-prompt API at any point in its runtime lifecycle.
- **Privilege confined to a tiny, fixed-purpose binary.** The Rust `root_helper` (Slices 3–4) is the only component that ever holds elevated capability, and only for a scoping window (capability dropped immediately after the needed `/proc` paths are opened), hardened by a kernel-enforced seccomp allowlist that makes the true CAP_SYS_PTRACE blast radius (full ptrace) practically unreachable even under a memory-safety bug or crafted input.
- **Two independent AND-gated opt-ins, not one.** OS-level grant (Slice 5, out-of-code, manual) AND Descartes config (`elevated.enabled:true`, default `false`, Slice 1) — both required, either absent degrades silently. This is enforced in Slice 2's core orchestrator and is the single most-tested code path in this plan (TDD items 1–2, 6, 11).
- **OS-level audience-scoping is independent of Descartes's own opt-in.** Primary mechanism: `root:descartes-provenance` group, mode `0750`, not world-executable, **plus a root-owned non-writable ancestor-directory chain** (Slice 2/5 — closes the dir-hijack local-privesc vector, where write access to a *containing* dir lets a local user swap the binary regardless of its own bits). Higher-privilege named `root_helper`/systemd fallback: its activation socket is equally scoped (`SocketMode=0660`, `SocketGroup=descartes-provenance`, Slice 5) so the root-running mechanism is never *looser* than the capability mechanism. Together these close the gap where a capability bit (or a root-run socket) alone would make the helper invocable by any local process regardless of Descartes's config.
- **The capability-bearing child is spawned with a minimal, explicit environment** (`{PATH:'/usr/bin:/bin'}`, no ambient `process.env` inheritance, Slice 2) — defense-in-depth against `LD_PRELOAD`/`LD_LIBRARY_PATH`-class injection, tested positively, not left to the kernel's secure-execution mode alone.
- **Untrusted input is never trusted.** Helper stdout is bounded, schema-validated, and echo-back scope-verified before any field is merged (Slice 2) — a malformed, oversized, or mis-scoped response falls back to the already-computed unprivileged `partial`/`0.4`/`missing_permission` result (owning UID preserved), never a distinct "partial" merge and never a collapse to the more-degraded `unable`/`0` shape.
- **`mechanism:"auto"` is deliberately narrow.** It only probes comparable-blast-radius mechanisms (`cap_sys_ptrace`/`helper_xpc`-equivalent); `root_helper` requires an explicit named operator choice, never reached by convenience default (Slice 2 test 11, re-checked at post-flight).
- **No installer overreach.** v1 ships with a documented manual grant and a real, tested graceful-degrade-when-absent path (Slice 5) — this plan does not build an automated root-requiring installer, matching the settled design's explicit scope boundary and the ground map's finding that no packaging pipeline exists to build one on top of.
- **CI privilege is disposable and bounded.** The only place anything runs as root in this entire plan is Slice 6's single new Buildkite step, inside an ephemeral per-job VM that is destroyed afterward, sharing the existing host-wide concurrency cap — never a developer machine, never a long-lived host.

## 5. Open questions (carried forward or newly raised by this plan)

1. **Does the CI Linux guest actually have passwordless sudo?** Unverified by anything in this repo (Layer B plan §9 open question, still open here) — first pre-flight item; blocks Slice 6 design specifics (which sudo invocation pattern, whether a dedicated sudoers rule is needed) until answered.
2. **Is the `ci-linux-arm64-rust-bazel` image's Rust toolchain actually usable as-is**, or does it need version pinning / a `rustup` step? Unverified — second pre-flight item.
3. **Real `ptrace_scope` value on the CI guest and on realistic target/production hosts** — Layer B §9 open question 5, still genuinely open; Slice 5/6 add a diagnostic, not a fix, and the finding should be recorded (not assumed 0/"classic") once Slice 6 actually runs.
4. **Is a compiled Rust binary acceptable build/packaging overhead for a v1 cut**, given zero Rust precedent in this repo today? Layer B §9 open question 4, restated — this plan proceeds on the operator's durable APPROVED decision, but the actual first-build experience (Slice 3) should be treated as the concrete answer, not assumed frictionless in advance.
5. **Linux x86_64 vs the only-available `linux-arm64` CI queue** — is arm64-only elevated-path CI validation sufficient for the stated Tier-1 platform claim elsewhere in project docs, or does a future x86_64 queue need to be requested from the `tart-ci` image owners? Not blocking Slice 6, but should be resolved before this plan is considered to have fully "validated in Linux CI" in the broadest sense the operator decision implies.
6. **Where should `crates/descartes-root-helper` actually live relative to the eventual `crates/descartes-core`/`crates/descartes-collector` HANDOFF sketch** — confirm the chosen layout (Slice 3) doesn't need reshuffling once those other crates are eventually scaffolded, since this is the first real Rust code and sets a precedent other future crates will follow.
7. **Named `root_helper` fallback's exact scoping *mechanism* (systemd socket-activation vs sudoers/polkit)** — Layer B §6(c) names both as options without picking one; Slice 5's documentation should pick one as the v1-documented path (systemd socket-activation is recommended here as the more auditable, narrowly-scoped default, given it colocates naturally with the existing daemon's systemd-user precedent noted in `linux-daemon-lifecycle-validation-brief.md`) but this is a design choice this plan flags rather than silently resolves without operator sign-off, since it's a new systemd unit shape not previously used in this repo for a root-running component. **Note (refine):** the *audience-scoping* of whichever mechanism is chosen is no longer open — it is now a fixed requirement (socket `SocketMode=0660`/`SocketGroup=descartes-provenance`, or an equivalently narrow sudoers/polkit rule, at least as tight as the primary binary's `0750 root:descartes-provenance`). Only the mechanism *selection* awaits operator sign-off.
