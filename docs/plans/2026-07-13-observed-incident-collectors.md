## 1. Safety spine (applies to every slice below unless a slice explicitly overrides — none do)

Restating `AGENTS.md`'s Safety Invariants against this plan's specifics:

- **Read-only by default.** Every new collector (Slices 1, 3) is fixed-argv `execFile` with timeouts, zero mutating flags, wrapped in the existing `timedEnvelope` fail-closed pattern (`tools/descartes-cli/src/tools/envelope.js:19-34`). Slice 2's evidence-freeze writes new Descartes-owned state but issues **zero new execFile calls** — it only invokes the already-registered, already-reviewed evidence-tool set.
- **Degrade-not-fabricate.** A collector that can't resolve a fact returns `unable`/`confidence:0`/`missing_permission` (or `partial`/`confidence:0.4` for the graceful cross-UID case, per `tools/descartes-cli/src/tools/provenance.js:458-461`, corrected path — must-fix 7a) — never a fabricated "0 sessions" or "no peers." This is asserted by a fixture per platform per collector, matching the existing per-platform clean/deleted-exe/public-bind/missing-permission fixture pattern from Slice 3/S3 of the parent roadmap.
- **No privilege escalation.** Slices 1–4 and 6 are same-UID, unprivileged, mirroring the existing provenance UID-scoping stance (`tools/descartes-cli/src/tools/provenance-identity.js:100-167`, corrected path — must-fix 7a: "asserts silence, not degraded confidence" for other-UID entities). If a slice's implementation turns up a genuine need for elevated reads (flagged as a live risk for Slice 3's VPN/peer status and Slice 5's event sources), the **only** acceptable model is the already-shipped S3-priv opt-in/audience-scoped/deny-by-default mechanism (`docs/plans/2026-07-11-s3-priv-elevated-read-path.md`) — flag it in the slice, do not design a new privilege mechanism in v0.
- **Sanitized, bounded output — hash-at-source is a hard requirement; this is NOT a property `sanitizeDiagnostics()` provides on its own (corrected claim, must-fix 3).** `sanitizeDiagnostics()` is a **syntactic** gate only (`^[A-Za-z0-9][A-Za-z0-9._:-]*$`, ≤64 chars) — it does not, and structurally cannot, distinguish a raw IP (`10.0.0.5`), a raw hostname (`host-01.internal`), a raw username (`thomas`), a charset-safe session name, or an ISO timestamp from any other charset-safe string: **all of these pass it unchanged.** Fact-record `attributes` in `fact-store.js` are only `String(value).slice(0,160)` — no sanitization at all — and `alert-store.js`'s `normalizeAlertRecord` plus `alert-intelligence.js`'s `compactAlert` (`:332-344`, corrected citation — Fable review of Slice 6, 2026-07-14; previously miscited as `:264`) pass stored `diagnostics` **verbatim** into the LLM prompt. Given this, every new identifier class this plan introduces (session names, peer hostnames/IPs/pubkeys, usernames from `who`/`last`) **must** be reduced to a fixed-length hex hash or a closed-enum bucket **in the translator/candidate-builder itself**, before the value ever reaches a fact record or a diagnostics payload — restated as a hard requirement with negative tests in Slices 1, 3, and 6 below. Entity keys that could carry operator-chosen free text (a tmux session named after a project, a VPN peer hostname) are additionally bounded via the existing `sanitizeIdentityString` helper — note this is **charset-substitution plus 64-char truncation, not hashing** (`diagnostics-sanitizer.js:88-99`, corrected description — must-fix 7d) — which bounds an entity key's shape but is not itself a confidentiality control and does not substitute for the hash-at-source requirement above.
- **Behind the kill switch.** All persisted fact-history and all learned-artifact-shaped emission (Slices 1, 3, 4, 6) is gated on `configDir/learned.json.enabled` (default `false`), reusing the exact structural-tick gate already wired in `daemon.js`. Slice 2 (evidence-freeze) is deliberately **not** gated on `learned.json` — it is an on-demand action like `descartes triage`, not an inference artifact, and has no draft/shadow/review-ready lifecycle because it captures facts rather than asserting a conclusion.
- **Day-1 no-storm.** Every new collector's first observation seeds a baseline/signature; nothing alerts on tick 1. Explicit test per slice.
- **Learned-artifact lifecycle where applicable.** Slice 4 (session-count baseline) is a `baseline` (statistical) family record under the existing `LearnedArtifact` envelope and `confidence_state:"provisional"|"established"` gate — no artifact self-promotes, no new lifecycle code. Slices 1, 2, 3, 6 do not mint learned artifacts themselves (1/3 are L0 facts; 2 is an action; 6 is a correlation rule evaluated fresh each tick, not a mined/promoted artifact in this pass).
- **The authority/containment boundary is unmissable.** See §0's table and Slice 7. No slice in this plan implements, stubs, or partially wires a mutating action against the monitored host.

## 2. Local testability model (why this is buildable on a macOS-only, no-VM dev machine)

Per the operator's dev-machine constraint (no `Virtualization.framework`, cannot run Linux/other VMs locally): every collector in Slices 0, 1, 2, 3, 4, 6 is **Node.js, `execFile`-based, and cross-platform-tested by dependency-injecting/mocking `execFile`'s output** — this is the established pattern already used by the existing dual-platform collectors (`network.js` exercises both the Linux `/proc/net/*` path and the macOS `lsof`/`scutil` path; `services.js` exercises both `systemctl` and `launchctl` branches) without ever touching a real Linux host. New collectors in this plan follow the identical pattern: real macOS execution is testable directly on this dev machine; the Linux branch is testable via mocked `execFile` fixtures pinned to real command output shapes, with CI (`.buildkite/pipeline.yml`'s `linux-arm64` job) providing the actual cross-platform execution proof, same division of labor as every existing collector.

**Only Slice 5 breaks this model** — if its spike concludes "go" on a privileged event-source build (auditd/eBPF/fanotify), that follow-on work is Rust, Linux-only, and privileged, requiring the exact `crates/descartes-root-helper` model already proven end-to-end in this repo: local `cargo check --tests` (cross-target, no execution) + real hardening/behavior tests gated to Linux CI (`tests/hardening.rs`, `tests/linux_resolution.rs` are both whole-file `#![cfg(target_os = "linux")]`). That is explicitly **out of scope for this plan** — Slice 5 itself ships only a decision document, never daemon-loop code.

## Slice 1 — Session-census L0 collector + fact-history translator

**locallyTestable:** true (both macOS and Linux branches, via mocked `execFile`; real execution testable directly on this dev machine since tmux/screen run natively on macOS).

**Recommended FIRST substantive slice of this milestone — see §3 for the validated justification.**

**Goal:** enumerate resident tmux/screen sessions (multiplexer, session name, attached/detached, window count, created-at) as a new L0 collector, and translate observations into the existing categorical fact-history so a session's presence — and its churn — is tracked over time, closing the "nothing tracks named sessions today, so a mass session-drop is invisible" gap named in the incident review. **Hard requirement (must-fix 6):** the collector captures a bucketed `created_at` attribute (see Files to add below) specifically so that a same-hour kill-then-resurrect — the actual shape of the motivating incident — surfaces as attribute churn (a session's `created_at` bucket changing between two ticks even though the session count is unchanged) rather than being invisible to a count-only hourly census.

**Files to add:**
- `tools/descartes-cli/src/tools/sessions.js` — new collector. `execFile('tmux', ['list-sessions', '-F', '...'])` and `execFile('screen', ['-ls'])`, wrapped in `timedEnvelope` exactly like `system.js`/`processes.js`. Degrade rule: neither binary present → `unable`/`confidence:0` (never "0 sessions"); binary present but genuinely zero sessions → `status:"ok"`, empty result (a real, distinguishable fact). **Hard requirement (must-fix 5):** bound the per-tick enumerated session count, mirroring `DEFAULT_DAEMON_PROCESS_LIMIT`'s existing cap pattern — a pathological session flood (e.g. a runaway script spawning thousands of tmux sessions) must not be allowed to blow past a fixed per-tick entity limit, because an unbounded fact-count from this one collector could evict *other* collectors' fact-history out of the shared 30-day/5MB `fact-store.js` cap and blind Slices 4/6 to unrelated signal. When the real count exceeds the cap, emit the cap's worth of entities plus an explicit `overflow:true`/truncated-count marker — never silently drop entities with no indication anything was truncated. **Nice-to-have:** `screen -ls` is known to exit with a non-zero status even when it succeeds and sessions exist — pin a fixture for this exit-code quirk so the collector doesn't misclassify a healthy `screen -ls` as a failure.
- `tools/descartes-cli/src/fact-translators.js` — add `factPointsFromSessionEvidence(envelope)`, mirroring `factPointsFromServiceEvidence` (`:25-49`). **Hash-at-source hard requirement (must-fix 3):** `sanitizeIdentityString` alone is charset-substitution + truncation, not a confidentiality control (see §1) — it bounds shape but does not stop an operator-chosen session name (a project name, a hostname pasted into a tmux session title, etc.) from surviving as recognizable text. This translator must reduce the session name to a fixed-length hex hash before it becomes part of `entity_key` (e.g. `entity_key = `${multiplexer}:${sha256(session_name).slice(0,16)}``, exact hash length TBD at implementation time) rather than merely pass it through `sanitizeIdentityString`. `attributes: {attached, window_count_bucket, created_at_bucket, ...}` — closed-enum/bucketed, never a raw free-text session name or raw timestamp reaching a fact record. **`created_at_bucket` (must-fix 6):** an hour-granularity (or finer, TBD) bucketed creation-time attribute, captured specifically so a same-hour kill-then-resurrect changes this attribute between ticks even when the session count alone would look unchanged.
- `docs/reference/collectors.md` — new `collect_sessions` entry.

**Files to change (registration seam — atomic, per the existing pattern):**
- `tools/descartes-cli/src/tool-policy.js` — add `collect_sessions` to `TRIAGE_TOOL_NAMES`. **Note (nice-to-have):** this makes the collector's raw, un-hashed session names visible to the operator-invoked `descartes triage` LLM session (unlike the persisted fact-history, raw triage-tool output is not translated/hashed before reaching that session) — this matches the existing consent posture for every other `TRIAGE_TOOL_NAMES` collector (triage is operator-invoked and already sees raw process/service data), so it is not a new exposure class, but it should be stated explicitly rather than left implicit.

### Slice 1 addendum (must-fixes 1 & 2, hard requirements) — census marker, exact-zero fold, partial-census skip

**Must-fix 1 — a mass session-drop to ZERO is currently INVISIBLE (the exact motivating incident).** `factPointsFromSessionEvidence` (`fact-translators.js:226-236`) returns `[]` when the collector succeeds with zero sessions. A zero-session structural tick therefore produces **no tick-group at all** — nothing for Decision 3's fold to see, so `session.count_drop` can never fire for a complete drop. Worse, a zero-fact tick is indistinguishable from a degraded collector (a `status:unable` envelope is filtered before it ever reaches the translator), so the fold cannot safely infer "0" without fabricating — that would violate degrade-not-fabricate (§1).

**FIX (a small Slice 1 addendum, folded in here — touches `fact-translators.js`, not a new Slice 4 file):** `factPointsFromSessionEvidence` emits one additional, always-present **census marker fact** on every successful `sessions` envelope, regardless of how many sessions were observed — including exactly zero. The marker:
- uses a fixed, non-hashed `entity_key` (e.g. `session.census-marker.v1`), the same pattern as `SESSION_OVERFLOW_ENTITY_KEY`;
- is excluded from Slice 1's per-tick entity cap/count, from churn detection, and from Decision 3's session count, exactly the way `SESSION_OVERFLOW_ENTITY_KEY` already is excluded;
- carries one closed-enum attribute, `census_state: "complete" | "partial"` (`"partial"` is must-fix 2's concern, immediately below — a plain census with no multiplexer failures is always `"complete"`).

**What this buys:** "one observation" (Decision 3) is redefined as *the number of non-marker `session.presence` points sharing a tick's `ts`, for a tick-group that also carries a census marker*. Zero non-marker points plus a `"complete"` marker is now a real, foldable **0** — not an absent tick-group.

**Pre-marker legacy history (recommended disposition):** tick-groups that predate this addendum carry no marker at all and are ambiguous — a markerless zero-count group could be a real 0 or simply the old translator's no-op. **Recommendation: skip markerless groups entirely** when folding (never fold them, never treat them as a zero, never error) — `last_folded_ts` still advances past them. This is an honest **~30-sample re-warm-up**: `confidence_state` cold-starts again from the first post-addendum marked tick-group, rather than silently mixing pre- and post-addendum semantics into one running statistic. State this explicitly as the deliberate, one-time cost of shipping the marker after Slice 1 already collected some real history — not a bug to fix later.

**DoD fixture correction (must-fix 1):** the "synthetic mass-drop" fixture (Decision 6, TDD surface and Definition of Done, below) is changed from a "near-zero-count tick" to an **EXACT-ZERO tick fixture** — a `"complete"`-marked tick-group with zero non-marker `session.presence` points — asserting the 0 is folded as a real sample and that `session.count_drop` fires against it. A "near-zero" (e.g. 1 remaining session) fixture may still exist additionally, but the exact-zero case is the one that maps directly to the motivating incident and must not be left implicit.

**Must-fix 2 — partially-degraded censuses currently fold as real observations.** `collectSessions` (`sessions.js:215,233`) reports envelope `status:"ok"` when **any** multiplexer succeeded — a per-mux `"unable"` (e.g. tmux ok, screen errored) is carried only on `result.multiplexers[]`, never persisted into the fact-history-facing shape. A tmux-ok/screen-errored tick therefore emits an undercount indistinguishable, at the fact-history layer, from a complete census — biasing the baseline low and able to false-fire `session.count_drop` on nothing more than a flaky `screen` binary.

**FIX:** the census marker introduced above carries `census_state: "partial"` whenever **any** multiplexer reported `"unable"` for that tick (a multiplexer reporting `"absent"` — genuinely not installed — does not count as partial; a host with no `screen` at all still has a complete census of what it actually runs). Decision 3's fold **skips** any tick-group whose marker is `"partial"`, exactly like an overflow tick: `last_folded_ts` advances past it, an observability counter increments (`skipped_partial_tick_count`, a sibling of `skipped_overflow_tick_count`), and the tick is never folded and never allowed to feed `session.count_drop`'s baseline or `session.churn`'s read.

**Fixture (must-fix 2):** a tick-group carrying a `"partial"` marker and an undercounted `session.presence` set asserts (a) the tick is excluded from the fold, (b) `last_folded_ts` still advances, (c) `skipped_partial_tick_count` increments, and (d) no `session.count_drop` candidate is derived from that tick's would-be `z_score`.

### Decision 1 — scope of v0 detection

**v0 ships exactly two signals, both over Slice 1's already-shipped `session.presence` fact-history, and nothing else:**

1. **Session-count deviation (`session.count_drop`)** — a statistical (Welford/EWMA z-score) drop in the number of resident sessions, catching the "mass termination" half of the incident.
2. **Session churn (`session.churn`)** — a deterministic fingerprint-diff on a persistent entity, catching the "kill-then-resurrect within one tick" half of the incident that count alone cannot see.

**Peer-count deviation (from Slice 3's `peer.presence` facts) is explicitly deferred to a fast-follow (Slice 4b), not v0.** Justification: (a) Slice 3's own design already deferred the *statistical* judgment on peer logins to "Slice 4" in the abstract, but folding it into *this* slice would double the math surface (a second independent Welford/EWMA metric needing its own fail-closed rule_id, e.g. `peer.count_drop`, reviewed at the same must-fix-1 bar); (b) peers have an extra dimensioning question session-counts don't — do WireGuard/SSH/`vpn_service` peers share one baseline or three (`source_type`-stratified)? — that has not been scoped and would need its own design pass, not a bolt-on; (c) this plan's own §3 sequencing logic ("bank the pattern on the simpler collector before extending it to a second one," the reason Slice 1 preceded Slice 3) applies identically here: prove the count+churn pattern on session data first, then extend the *same* module's pattern to peers once it's shipped and observed, rather than shipping two new statistical families in one slice. Recommendation: pick up peer-count deviation as Slice 4b immediately after Slice 4 ships and has accumulated real signal.

**Hard dependency:** Slice 1's session fact-history must exist and accumulate real samples before a count baseline is meaningful — this slice ships code + tests against synthetic fixtures immediately, but the first real count-deviation alert on live data won't fire until enough real hourly samples accumulate past `min_sample_count` (cold-start honesty, identical posture to every other baseline in the parent roadmap). Churn has no such warm-up: it can fire the first time any entity is observed twice with a changed fingerprint (see Decision 3).

**Explicit check before starting:** if the parent roadmap's general S10 `baseline-store.js` has shipped by the time this slice is picked up, **consume it directly** instead of duplicating Welford/EWMA math in a new file — confirmed by search (`find tools/descartes-cli -iname "*baseline*"`) that no such file exists in the tree today, so this slice's own scoped math implementation is the correct default, not a redundant parallel engine. If S10 lands first, retarget `computeSessionBaselineCandidates` to read/write through `baseline-store.js`'s API instead of `session-baseline.js`'s own store, keeping the `dimension_bucket` scoped to `"session.count"`/`"global"`.

### Decision 2 — fail-closed namespace (HARD, must-fix 1) — RESOLVED

**Rule_ids, pinned:** `session.count_drop` and `session.churn`.

**Confirmed against the shipped code, not merely asserted:** `alert-intelligence.js:394-406`'s `classifyAlertNamespace` is a closed, ordered `startsWith` chain — `learned.` → `daemon.`/`system.`/`disk.` → `constraint.` → `provenance.` → `baseline.` → `identity.` → (fallback) `{ namespace: undefined, hardExcluded: false }`. This is must-fix 1's option 1 (the plan's own "v0 default, recommended" option), now confirmed correct by direct code inspection rather than left as an untested assumption — **option 2 (the `baseline.session.` classifier-branch alternative) is not pursued in this slice**; it remains recorded above only as an available future alternative if a deliberate namespace decision is later made to enable LLM adjudication for this family.

**The alert is NOT silently dropped by this design** — `session.count_drop`/`session.churn` candidates are still generated, persisted to `alerts.json`, and eligible for the existing local notification path exactly like any other alert; only S13's LLM adjudication is withheld (`unknown_namespace`), which is the intended, safer posture for a brand-new, unreviewed statistical family whose false-positive rate is not yet known in production.

### Decision 2b — deterministic, non-LLM local delivery (must-fix 3, hard requirement)

**The gap:** Decision 2 correctly keeps `session.count_drop`/`session.churn` out of LLM adjudication (`unknown_namespace`), but as drafted these alerts could **never actively notify the operator**. `deliverNotificationDecision` is reached only by (a) an ELIGIBLE, LLM-adjudicated alert, (b) the budget-exhausted signal, or (c) the `descartes alerts` CLI test command. A `session.*` alert is `unknown_namespace` and is filtered out *before* delivery (`alert-intelligence.js:485-490`). Concretely: this milestone's first alerting slice — including a CRITICAL mass-drop, the exact motivating incident — would be visible **only** if an operator proactively ran `descartes alerts`. That is not "alerting," it is a log an operator has to remember to go read.

**FIX (preferred option a):** add a **deterministic, non-LLM local delivery branch**, following the existing no-LLM precedent already shipped for `emitBudgetExhaustedSignal` (`alert-intelligence.js:235-251` — "straight through `deliverNotificationDecision`, NEVER through the LLM"). Concretely:
- the branch fires **only** for the fail-closed session rule_ids (`session.count_drop`, `session.churn`) — scoped by an explicit rule_id allowlist, not a general unknown-namespace bypass, so it cannot silently swallow some future, unrelated fail-closed namespace;
- the decision handed to `deliverNotificationDecision` is **hand-built and deterministic** — never a session, never an LLM call, never `enableTools`;
- it reuses the `notification_due_ids`/cooldown state that `applyAlertCandidates` already computes for every candidate (including these two rule_ids) — no new cooldown machinery;
- the delivered body is **counts/hash-only** — `observed_count`/`mean_before`/`z_score`/`confidence_state` for `session.count_drop`, `entity_key`/`prior_fingerprint`/`current_fingerprint` (all hashes) for `session.churn` — never a raw session name, matching the diagnostics shape already specified in Decision 4.

**Correction, load-bearing (must-fix 3):** `applyAlertCandidates` stamps `last_notified`/`cooldown_until` on a candidate **even when nothing is actually delivered** downstream (e.g. today, for an `unknown_namespace` alert that never reaches `deliverNotificationDecision` at all). Do **not** read the presence of `last_notified`/`cooldown_until` as evidence that a notification was delivered — that stamp reflects only that the candidate was *processed* by `applyAlertCandidates`, not that any human-visible delivery happened. This new branch's own tests must assert on an actual call into (or mock of) `deliverNotificationDecision`, not merely on the cooldown/`last_notified` fields.

**Files-to-change addition:** `alert-intelligence.js` gains the new deterministic delivery branch (near `emitBudgetExhaustedSignal`, `:235-251`) plus a small rule_id allowlist constant (e.g. `DETERMINISTIC_LOCAL_DELIVERY_RULE_IDS = [SESSION_COUNT_DROP_RULE_ID, SESSION_CHURN_RULE_ID]`, importing the constants from `session-baseline.js` rather than restating the string literals).

**TDD addition:** a fixture asserting a due `session.count_drop` candidate (and separately `session.churn`) reaches `deliverNotificationDecision` through the new deterministic branch with zero LLM/session construction; a fixture asserting the existing cooldown is respected (a second due tick within the cooldown window does not re-deliver); a fixture asserting the delivered body contains only counts/hashes, never a raw session name; a fixture explicitly asserting that `last_notified`/`cooldown_until` being stamped is NOT, on its own, treated as proof of delivery — the test asserts the delivery call happened, not just the stamp.

**DoD addition:** the deterministic local delivery branch ships, is scoped to exactly the two session rule_ids, respects the existing cooldown, delivers counts/hash-only bodies, and its tests assert on actual delivery rather than on `last_notified`/`cooldown_until` alone.

### Decision 3 — baseline math + confidence_state gate

**One dimension bucket in v0:** `dimension_bucket: "global"` — no per-time-of-day or per-entity stratification (that is the parent roadmap's Slice 12 territory). The persisted record mirrors the parent roadmap's own baseline-record shape (§7): `{ metric_name: "session.count", dimension_bucket: "global", time_bucket: null, stats: { count, mean, m2, variance, stddev, ewma, ewma_variance, min, max }, confidence_state: "provisional"|"established", first_seen, last_updated, sensitivity: "operational", schema_version }`, plus Slice-4-specific additive fields: `last_folded_ts` (an ISO timestamp checkpoint, see below), `last_observation: { ts, count, z_score, has_overflow }`, and `skipped_overflow_tick_count` (observability counter, see overflow handling below).

**The fast-tick-vs-hourly-fold hazard (resolved, load-bearing):** `computeSessionBaselineCandidates` is invoked on **every** daemon tick (~60s, via `extraCandidates`), but new `session.presence` facts only land once per hourly structural tick (`daemon.js`'s structural-tick block). A naive "read latest fact, fold into Welford" on every call would fold the *same* hourly observation into the accumulator up to ~60 times before the next real observation lands, catastrophically biasing `mean`/`variance` toward whichever observation happened to be current when the daemon last restarted its polling cadence. This is prevented by:
- **Tick-grouping via shared `ts`, now marker-aware (must-fixes 1/2):** every fact point emitted by one structural tick shares the exact same `ts` string (confirmed: `daemon.js`'s `const ts = options.ts ?? new Date().toISOString()` is passed identically to `factPointsFromSessionEvidence(structuralEvidence, { ts })` and every sibling translator call within one `runDaemonIteration` invocation). "One observation" = the count of non-marker `session.presence` fact points sharing one such `ts`, for a tick-group that also carries a `"complete"` census marker (must-fix 1) — a tick-group with a `"partial"` marker (must-fix 2), no marker at all (pre-addendum legacy history), or the `SESSION_OVERFLOW_ENTITY_KEY` marker is handled by the skip rules below, never folded as a plain observation.
- **`last_folded_ts` checkpoint — scope corrected (must-fix 5): gates observability bookkeeping and the recompute trigger only, NOT an incrementally-folded statistic.** `computeSessionBaselineCandidates` reads `session.presence` fact-history via `readFactPoints` bounded to a generous window (`DEFAULT_BASELINE_FACT_WINDOW_MS`, e.g. 31 days — comfortably covering `fact-store.js`'s own `DEFAULT_FACT_RETENTION_MS` 30-day ceiling), groups by `ts`, and identifies tick-groups newer than `state.last_folded_ts` purely to decide (a) whether the windowed recompute (below) needs to re-run at all this tick, and (b) whether to advance past a skipped (overflow/partial/markerless-legacy) tick-group and increment its observability counter exactly once, not once per fast tick. In steady state this is exactly one new tick-group per hour; every other (fast) tick finds zero new groups, performs no recompute, and writes nothing. **Backward-clock-step note (nice-to-have v):** because the gate is strictly-newer-than (`ts > state.last_folded_ts`), an NTP step backward on the host could produce a tick-group whose `ts` is not strictly newer than the checkpoint, which would silently skip that tick-group's fold/recompute-trigger with no error surfaced. Document this as a known, accepted limitation (a backward clock step is already an unusual host event) rather than leaving it undiscovered — a future hardening pass could compare with a small tolerance instead of a strict `>`, but that is not required for v0.
- **Welford/EWMA recompute is WINDOWED, not incrementally accumulated (must-fix 5, hard requirement — corrects the design as originally drafted):** the originally-drafted design persisted an accumulator (`count`/`mean`/`m2`) folded once per new tick-group and then kept **forever**, growing without bound — this directly contradicted the plan's own "bounded by fact-store's 30-day retention" claim (retention bounds what is *re-foldable*, not a persisted running total, which never shrinks on its own). Two concrete consequences of the unbounded design: the gradual-drift guard would keep weakening as `n` grew arbitrarily large (a slow multi-day decline could eventually false-alarm once the accumulator's inertia got large enough), and a legitimate regime change (the operator now intentionally runs 8 sessions instead of 20, going forward) would keep `session.count_drop` active for months, since the old high-count era's samples never left the average. **FIX:** on every tick where a recompute is triggered (per the `last_folded_ts` gate above), `foldWelford`/`updateEwma`'s inputs are **recomputed fresh from scratch** over every complete (non-overflow, non-partial, per must-fixes 1/2) tick-group's non-marker session count currently inside the 31-day read window — never an incremental fold onto a persisted running total. This makes the plan's own "bounded by 30-day retention" claim literally true (the stats *are* the window), makes the mechanism auto-forgetting (a tick-group that ages out of the window stops contributing on the very next recompute, with no explicit decay math needed), and removes the `last_folded_ts` double-count hazard for the stats entirely — recomputing the same window twice is idempotent by construction, so there is no possible double-fold to guard against. Cost: O(up to ~720) per hourly recompute, trivial (the module already reads the full window every tick for churn's own purposes, per this Decision's churn section below, so this adds no new I/O class, only more arithmetic over data already in memory). `state.stats.count` in the persisted record is therefore reinterpreted as "the number of complete tick-groups currently inside the window," not a lifetime total — it can and will decrease as old tick-groups age out, which is expected and correct.
  - **Alternative considered, not selected as the default:** keep the incremental accumulator but add explicit forgetting (a capped effective `n`, or exponential decay weighting older folds down). Rejected as the v0 default because it reintroduces a tuning parameter (decay rate/cap) with no incident-motivated value yet, whereas the windowed recompute has no new parameter and directly matches the plan's already-stated retention claim. Recorded here so a future revision can revisit if O(720) per hourly recompute ever stops being trivial.
- **z-score computed against the window EXCLUDING the tick-group under test (renamed from "z-score-before-fold," same self-dampening-avoidance intent, must-fix 5 wording correction):** for the latest tick-group, compute `mean_before`/`stddev_before` from the windowed recompute over every OTHER complete tick-group currently in the window (i.e. the window as it stood immediately before this tick-group is included), then `z = (observed_count - mean_before) / max(stddev_before, STDDEV_FLOOR)`, record that as `last_observation.z_score`, and only then does the persisted `stats` snapshot reflect the recompute that includes the new tick-group. Computing z against a window that already includes the point under test would dampen genuine anomalies — this ordering is a hard requirement, not a style preference, and must be asserted by a dedicated unit test (see TDD surface), now phrased for the windowed-recompute design rather than the retired incremental-fold design.
- **Overflow-tick handling (degrade-not-fabricate, unchanged in substance):** if a tick-group carries the `session.overflow-marker.v1` entity (i.e. `envelope.result.truncated` was true that tick — see Slice 1's must-fix 5), the tick's true count exceeds `DEFAULT_SESSION_ENTITY_LIMIT` and only a bucket (`total_count_bucket`) is known, not an exact integer. This tick-group is excluded from the windowed recompute entirely (an inexact/bucketed value must never masquerade as an exact sample in a window-derived mean/variance) — `last_folded_ts` still advances past it (so it is not retried forever) and `skipped_overflow_tick_count` increments for observability. A genuine session-flood is therefore visible via the overflow marker itself (already a Slice 1 concern) without corrupting the count-drop baseline.
  - **Overflow-tick effect on an already-active `session.count_drop` (plan-text nice-to-have ii, decided + tested):** an overflow tick occurring while `session.count_drop` is already active neither implies the drop resolved nor suppresses the existing candidate. Because Decision 3's re-emission-every-tick discipline re-derives the candidate fresh from `state` on every call, and an overflow tick contributes nothing to that state (it is excluded from the recompute, per above), the existing candidate keeps re-emitting unchanged from the last real complete-tick-group's `z_score` until a real (non-overflow, non-partial) tick-group either confirms recovery or updates the observation. `last_observation.has_overflow` is set to `true` for that tick purely as an observability flag — it is never read by the trigger/recovery logic itself. A dedicated test asserts an overflow tick occurring during an active `session.count_drop` leaves the candidate's `z_score`/severity unchanged from the prior real observation.
- **`STDDEV_FLOOR` guard (default `0.5` sessions, PROVISIONAL — see must-fix 7 below):** prevents a near-zero windowed variance (e.g. a host that has run exactly N sessions for weeks, so every complete tick-group in the window has the same count) from making the z-score denominator blow up and firing on trivial ±1 fluctuations. A named, fixture-tested constant, not a magic number folded into the formula.

**Deviation trigger:** `session.count_drop` fires when `confidence_state === "established"` **and** `z <= -DEVIATION_SIGMA` (default `DEVIATION_SIGMA = 3`) **and** `observed_count < mean_before` (redundant given a negative z, kept as an explicit defense-in-depth guard against a formula regression). Severity escalates from `"warning"` to `"critical"` at a sharper threshold, `z <= -CRITICAL_SIGMA` (default `5`) — baked into v0 rather than deferred, since it costs nothing beyond a second constant and no new consent/namespace machinery. **A symmetric positive-z "mass session appearance" spike is explicitly NOT in v0 scope** — the motivating incident was a drop, not a burst; noted as a trivial fast-follow (drop the sign guard) rather than designed now, to keep v0's trigger surface minimal and reviewable.

**PROVISIONAL constants flag (must-fix 7, hard requirement — do not read `DEVIATION_SIGMA=3`/`CRITICAL_SIGMA=5`/`STDDEV_FLOOR=0.5` as settled):** these three constants are placeholder defaults chosen to unblock shipping v0, not tuned values. **Flag them explicitly as PROVISIONAL — "tune after real data" — the same posture the parent roadmap already takes toward its own untuned thresholds.** **Low-variance severity-inflation risk, stated concretely:** with a steady baseline (`stddev` at or near the `STDDEV_FLOOR`), the floor makes the absolute trigger `3 × 0.5 = 1.5` sessions and the critical tier `5 × 0.5 = 2.5` sessions — so on a host that has stably run **5 sessions** for weeks (`stddev` near zero, floored to `0.5`), closing just **3** sessions (a routine end-of-day cleanup, not an incident) crosses `z <= -5` and fires **CRITICAL**. The day-1/cold-start no-storm guarantee (§1, this Decision) still holds — this is not a storm-on-tick-1 problem, `confidence_state` must reach `established` first — but it is a **credibility risk**: an operator who gets a CRITICAL alert for closing 3 of 5 routine sessions will reasonably distrust the signal going forward. **Nice-to-have, recorded for a future pass, not required for v0:** a min-absolute-drop guard (e.g. require `mean_before - observed_count` to exceed some fixed floor) or a fraction-of-mean guard (e.g. require the drop to exceed some fraction of `mean_before`, not just a sigma multiple) on the CRITICAL tier specifically, so a small-baseline host's routine variation doesn't reach the sharper severity tier on sigma alone. Left as a nice-to-have rather than a v0 requirement because it needs real operational data to tune sensibly — exactly why these three constants are flagged provisional in the first place.

**`confidence_state` gate:** `"provisional"` while `state.stats.count < min_sample_count` (default `30`, matching the parent roadmap's stated S10 default — `docs/plans/2026-07-09-self-learning-stratified-monitoring.md:77/138/176`); `"established"` once `count >= 30`. **No `session.count_drop` candidate is ever emitted while `provisional`** — this is the day-1/cold-start no-storm guarantee for the count signal. Unlike the parent roadmap's general metric baselines (which keep a *fixed* threshold active during cold-start as a safety net), session-count has no pre-existing fixed threshold to fall back to — nothing tracked named sessions before Slice 1 — so cold-start here means genuinely **no** session-count alerting until 30 real hourly samples accumulate (a little over a day of continuous structural-tick operation). State this plainly rather than imply a safety-net threshold exists where it doesn't.

**Gradual-drift guard, mechanism made explicit (must-fix 5 correction: this claim is now actually true under the windowed-recompute design, not merely asserted):** because `z` is computed against the windowed recompute (mean/stddev derived over up to ~720 complete hourly tick-groups currently inside the 31-day read window, per must-fix 5 above — genuinely bounded now, since the stats ARE the window, not a persisted running total that outlives it), a slow multi-day decline shifts `mean` (and generally widens `stddev`) incrementally alongside each new sample — at any given tick, the next sample is close to the *already-adapted* windowed mean, so no single tick produces a large z-score the way one true mass-drop tick does. This is the concrete mechanism (not an assertion) that satisfies the "z-score against trailing distribution, not naive tick-over-tick delta" requirement.

**Gradual-drift fixture, pinned to a stated realistic rate (must-fix 5, hard requirement):** the synthetic gradual-drift fixture (TDD surface below) must state and use a concrete, realistic decline rate rather than an arbitrary "small per-tick decrement" — e.g. a session count declining by roughly 1 session per day over 2–3 weeks (a plausible organic falloff, not a proxy for a real incident) — and assert no false alarm at that stated rate. Pinning the rate makes this fixture falsifiable and reviewable rather than a vague qualitative claim.

**Regime-change fixture, new (must-fix 5, hard requirement):** a fixture establishes a stable baseline at count `N` for `>= min_sample_count` synthetic hourly tick-groups, then shifts to a **sustained, legitimate** new steady state at count `M < N` (e.g. the operator now intentionally runs fewer sessions going forward) for a further run of tick-groups, and asserts that `session.count_drop` **eventually recovers without any operator action** once enough post-shift tick-groups have entered the window and the windowed mean/stddev have adapted to the new regime — proving the windowed-recompute design (unlike the retired unbounded-accumulator design, under which the old high-`N` era would never leave the average) actually self-corrects for a legitimate regime change rather than alerting indefinitely.

**Re-emission every tick (load-bearing, confirmed against `alert-store.js`):** `applyAlertCandidates` (`alert-store.js:220-263`) marks any currently-`active` alert whose id is **absent** from the current tick's merged candidate array as `"recovered"` — this is how the existing `extraCandidates` sources (e.g. `computeProvenanceWarningCandidates`) already behave, and `computeSessionBaselineCandidates` must follow the identical discipline: it re-derives and re-emits the `session.count_drop` candidate **on every call**, for as long as `state.last_observation.z_score` still satisfies the trigger — never only once at fold time. Concretely: the candidate is built fresh from the currently-persisted `state` on every invocation (a cheap read, no re-fold), exactly mirroring how `computeProvenanceWarningCandidates` reads "latest known state" and maps it to a candidate fresh each tick.

**EWMA's role, scoped explicitly:** `ewma`/`ewma_variance` are computed and persisted (matching the shared parent-roadmap baseline-record schema, for forward compatibility with `descartes learned`-style tooling and a future S10 migration) but are **not** consumed by v0's trigger formula — the trigger is Welford-mean/stddev z-score only, per the plan's own explicit "not naive tick-over-tick delta" requirement. Designing a second, EWMA-driven trigger without an incident-motivated justification is out of v0 scope; noted here so the omission reads as a scoping decision, not an oversight.

**Churn detection (session.churn), a separate and simpler mechanism — no persisted state, no confidence_state gate:** for each `entity_key` with two or more `session.presence` fact-history points within the read window (excluding the overflow-marker entity and, per must-fix 2, any point belonging to a `"partial"`-marked tick-group), compare the two most recent points' `created_at_fingerprint` attribute. `session.churn` fires when both fingerprints are defined, neither is the literal string `"unknown"`, they differ, **and (must-fix 4, hard requirement, recency bound) the newer of the two points belongs to the latest — or one of the last K, K small (v0 default K=1) — COMPLETE tick-group(s)** (a complete tick-group per must-fixes 1/2: it carries a census marker, and that marker is neither `"partial"` nor absent). This is a **pure function of already-persisted fact-history** — recomputed fresh every tick directly from `readFactPoints`, exactly like `provenance-warnings.js`'s `reduceLatestProvenanceWarnings` pattern — so it naturally clears/recovers once a third, stable observation makes the "two most recent" pair no longer span the change; no separate persisted churn state is needed. A brand-new `entity_key`'s first-ever observation cannot churn by construction (nothing to diff against yet), which is itself this signal's day-1 no-storm guarantee — no explicit `min_sample_count`-style gate is needed for churn.
- **Documented v0 limitation:** `screen` sessions never expose a creation time — `fingerprintCreatedAt` returns `"unknown"` whenever `created_at_epoch_seconds` is non-finite, which is always true for `screen -ls` output (confirmed in `fact-translators.js`'s `buildSessionFactPoint`) — so churn detection is effectively **tmux-only** in v0. State this explicitly as a known scope limitation of the underlying collector, not a gap discovered later as a surprise.

**Recency bound rationale (must-fix 4, why this is a hard requirement, not a nice-to-have):** without a recency bound, "compare the two most recent points" fails two concrete ways:
1. **Stale-pair persistence.** An entity that churns and then **disappears** (killed, never resurrected) never gets a third observation to retire the pair. Its churned pair stays "the two most recent" for that `entity_key` until the entire pair ages out of the 31-day read window — so `session.churn` reads as **active for close to a month** for a session that has been gone the entire time. The recency bound fixes this: once the entity is absent from the latest complete tick-group, the newer point of its pair is no longer "in the latest complete tick-group," so the candidate stops firing immediately, without needing the entity to reappear.
2. **Upgrade-day history replay.** On the first tick after Slice 4 deploys, Slice 1's already-shipped weeks of fact-history are scanned for the first time. Every historical fingerprint change already sitting in that history — including every reboot, which recreates all tmux sessions with new `created_at_fingerprint`s — would fire immediately as churn, producing an **upgrade-day storm** that defeats the plan's own day-1 no-storm invariant (§1). The recency bound closes this too: a churned pair buried in old history has neither point in the *latest* complete tick-group, so it is silent on first run; churn only starts firing going forward, for changes observed from that first post-deploy tick onward.

**Non-fire fixtures (must-fix 4, hard requirement, in addition to the existing churn fixtures in Decision 6's TDD surface):**
- **Stale pair:** an entity churns between tick T-5 and T-4, then is absent from every tick through the latest (T). Asserts `session.churn` does **not** fire (the pair is stale — its newer point is not in the latest complete tick-group), even though naively "the two most recent points for that entity" still differ.
- **Pre-existing-history-at-first-run:** a fact-history fixture pre-populated with several weeks of already-differing fingerprints (simulating Slice 1 having run before Slice 4 was deployed, including a synthetic reboot-shaped fingerprint change) asserts that the **first** call to `computeSessionBaselineCandidates` after this history exists emits **zero** `session.churn` candidates — proving no upgrade-day storm — and that churn only begins firing for entity changes observed in tick-groups from that first call onward.

### Decision 5 — state store

`tools/descartes-cli/src/session-baseline.js` exports `resolveSessionBaselineStorePaths(descartesPaths)` returning `{ dir: path.join(descartesPaths.stateDir, "learned"), storeFile: path.join(dir, "session-baseline.json") }` — a sibling of `signatures.json`/`constraints.json` under the same `stateDir/learned/` directory, matching `provenance-store.js`'s `resolveSignatureStorePaths` convention exactly (`provenance-store.js:92-95`).

- **Read:** `loadSessionBaselineStore` reuses the same ENOENT-tolerant + corrupt-tolerant helper pattern as `provenance-store.js`'s `readJsonFile` (`provenance-store.js:257-270`): missing file → fresh/empty baseline state (`count:0`, `confidence_state:"provisional"`); malformed JSON → `corrupt:true` marker, state **resets to a fresh baseline** rather than throwing out of a daemon tick (mirrors `constraint-store.js`'s `loadLearnedConfig` fail-closed-to-default posture for corrupt JSON). A corrupted state file must never take down alert evaluation; the cost of resetting is an honest, documented cold-start (30 real samples before `established` again), not a silent data-loss surprise — `corrupt:true` should be surfaced the same way `loadLearnedConfig` surfaces it.
- **Write:** atomic tmp+rename, `0o600`, directory `mkdir(..., { recursive: true, mode: 0o700 })` — identical to `provenance-store.js`'s `writeSignatureStore` (`provenance-store.js:296-302`) and every other store in this codebase.
- **Update cadence (corrected, plan-text fix i):** the small `session-baseline.json` **state** file is read every tick (cheap) and only **written** on ticks where a new tick-group was actually recomputed/folded (Decision 3) — roughly once per hour. **Correction:** the earlier "not the multi-hundred-KB fact-history" phrasing here was wrong and must not be read as license to skip reading fact-history on fast ticks. Decision 3 already requires `readFactPoints` on **every** tick regardless, for two independent reasons: (a) churn detection is a pure function recomputed fresh from `readFactPoints` on every single call, with no persisted state of its own; (b) must-fix 5's windowed Welford recompute needs to identify whether a new complete tick-group has landed since `last_folded_ts`, which likewise requires reading the fact-history's recent tick-groups every tick, not just at fold time. So `readFactPoints` (the actual multi-hundred-KB-scale read) is **not** skipped on fast ticks — only the small `session-baseline.json` store's **write** is skipped on ticks that found nothing new to fold. An implementer must not "optimize away" the churn/window-recompute `readFactPoints` call on fast ticks on the mistaken belief that this sentence licensed doing so.
- **Relationship to the parent roadmap's `LearnedArtifact` envelope (clarifying an ambiguity in Section 1's safety spine):** this record is a **statistical/baseline-family artifact** per the parent roadmap's data-shape taxonomy (§7), but it does **not** enter the `draft → shadow → review-ready → active` promotion state machine that governs *constraint*-family artifacts (Layer A, S6c/S7) — per the parent roadmap's own Slice 10 framing, a statistical baseline gates purely on `confidence_state` reaching `min_sample_count`, with no human `approve` step required to go from `provisional` to `established`. This is an existing-precedent-consistent design choice, not a new invention introduced by this slice.

## Slice 4b — Peer-count deviation (`peer.count_spike`)

**Created:** 2026-07-14 — deepened design for the fast-follow explicitly deferred from Slice 4's Decision 1 ("Peer-count deviation ... is explicitly deferred to a fast-follow (Slice 4b), not v0"). Mirrors the shipped Slice 4 session-count baseline (`session-baseline.js`, `cf0813f`) over the shipped Slice 3 `peer.presence` fact-history (`fact-translators.js`, `91168d1`). **Design pass only — not yet implemented, not yet Fable-reviewed** (flagged for review before implementation — every prior slice with real alert-emission surface got a dedicated Fable pass before or immediately after shipping; this one should not be an exception).

**locallyTestable:** true (synthetic fixtures; no real accumulated peer traffic needed for code/tests — mirrors Slice 4's own cold-start note; real alerting on live data still needs real elapsed hourly ticks, same posture).

**Goal:** detect the incident's peer-side signature — a mass, odd-hour peer-login **burst** — as a statistical deviation over Slice 3's already-shipped `peer.presence` fact-history, reusing Slice 4's exact windowed-Welford pattern rather than inventing new math. Unlike Slice 4's session-count signal, this is a **spike** (an increase), not a drop: the motivating incident's peer half was unattributed/odd-hour logins *appearing*, not disappearing, so this slice does not inherit Slice 4's drop-to-zero-invisibility hazard (must-fix 1) — see Decision 2 below for why that matters and what is still deferred.

### Decision 1 — scope of v0 detection: spike, not drop; what "one observation" means

**v0 ships exactly one signal:** `peer.count_spike` — a Welford/EWMA z-score **increase** in the number of currently-observed peers, mirroring `session.count_drop`'s math with the sign and trigger direction flipped.

**Justification for spike-over-drop as the v0-defining signal (not merely "spike is easier"):** the todo's originating incident (`todos/2026-07-09-self-learning-stratified-monitoring.md:114-129`) describes *unattributed VPN/SSH logins at odd hours* co-occurring with the session mass-termination — an **appearance** of peers, not a disappearance. A drop-shaped detector would target a different (real, but not-this-incident) failure mode: a VPN/SSH access channel going dark (e.g. a WireGuard interface silently stops handshaking, or all remote operators are unexpectedly absent). That is a legitimate future signal, but it is not what this milestone's motivating incident needs, and — per Decision 2 below — it comes with a real correctness cost (a census-marker addendum) that a pure spike detector does not.

**What "one observation" means (v0):** the count of `peer.presence` fact points sharing one structural tick's `ts`, **restricted to points whose `attributes.presence_state === "observed_active"`** (`fact-translators.js:395`), excluding the `PEER_OVERFLOW_ENTITY_KEY` marker (`fact-translators.js:326`, already shipped). Peer-status is registered on the same hourly structural cadence as sessions — confirmed: `daemon.js:252-253` registers `sessions` and `vpn-peer-status` back-to-back in `collectStructuralEvidence`'s `activeCollectors` map, so every peer fact emitted in one daemon iteration shares the identical `ts` as every session fact from that same iteration, per the same mechanism Slice 4's own ts-cohesion test already proves for `session.*`.

**Why `observed_active`-only, and not also `observed_historical` (a deliberate, reasoned exclusion — not an oversight):** Slice 3's SSH source blends two sub-sources per tick — `who` (currently logged-in, tagged `observed_active`) and a **bounded** `last -n N` (recent login history, tagged `observed_historical`), deduplicated against `who` by `(remote_user, remote_host)` so the same login is never double-counted (confirmed: `vpn-peer-status.js:383-419`'s `dedupeKey`/`seenSsh` logic). `last -n N` returns the **N most recent** records regardless of a time window, not "records from the last hour" — so once N history exists, its count saturates near `min(N, available history)`: a burst of new logins causes **turnover** (the oldest of the N pushed out), not necessarily a **count increase**. Folding `observed_historical` into the spike count would therefore make the count numb to exactly the burst this signal exists to catch, once the window is already full — a subtle but real correctness gap, not merely a stylistic preference. `observed_active` (currently-connected peers — WireGuard peers, `who`-visible SSH sessions, connected `vpn_service` entries) has no such saturation hazard and is the direct peer-side analogue of Slice 4's "currently resident session count."

**Documented v0 limitation (the peer-side analogue of Slice 4's tmux-only-churn limitation, recorded rather than discovered later):** a peer that connects and fully disconnects within a single hourly tick window is invisible to `observed_active`-only counting — the exact "kill-then-resurrect within one tick" shape Slice 4 solved for sessions via `created_at_fingerprint` churn detection. Peers have **no equivalent per-entity churn signal in v0** (Slice 3 emits no peer-side creation/reconnection fingerprint), so a `peer.churn`-style detector is **explicitly out of v0 scope** — it would need its own design pass (what would "peer identity churn" even mean: a reused ephemeral SSH port? a WireGuard peer that drops and re-handshakes? neither maps cleanly onto Slice 1's tmux `created_at_fingerprint` precedent). Recorded here as a second, distinct deferred follow-up, separate from Decision 2's drop/census-marker follow-up.

**Dimension stratification: one global bucket in v0, not per-`source_type` (WireGuard/SSH/`vpn_service`).** Mirrors Slice 4 Decision 1's own "one dimension bucket in v0" choice exactly. Slice 3's own design explicitly flagged "do WireGuard/SSH/`vpn_service` peers share one baseline or three?" as an unscoped question; v0 resolves it to **one** (a single `dimension_bucket:"global"` count, summed across all `source_type`s) for the same reason Slice 4 stayed single-bucket: three independent confidence-gated baselines (three separate `min_sample_count` warm-ups, three separate PROVISIONAL-sigma tuning surfaces) is real added complexity with no incident-specific justification yet. **Known dilution risk, stated plainly (PROVISIONAL, same class as Slice 4's must-fix-7 sigma-constant flag):** if WireGuard peers are a large, stable, near-constant population and SSH peers are the volatile/bursty ones, pooling both into one count could dilute a pure-SSH burst's z-score relative to a `source_type`-stratified baseline. Recorded as a fast-follow (stratify by `source_type` once real data shows dilution is actually a problem), not a v0 blocker.

**Odd-hour weighting (`login_hour_bucket`): deferred, not v0.** Slice 3 already captures `login_hour_bucket` (`fact-translators.js:396`) as a closed-enum attribute on every peer fact, specifically so a future statistical pass could consume it — but stratifying (or weighting) the baseline by hour-of-day means either 24 separate confidence-gated sub-baselines or a smarter partial-pooling scheme, neither justified without evidence the flat (hour-unaware) baseline actually under-detects in practice. **Stated limitation:** a flat, hour-unaware baseline may under-detect a burst that is anomalous *for its time of day* but not anomalous against the whole day's variance (e.g. a host that routinely sees 8 concurrent SSH peers at 2pm could absorb a 5-peer 3am burst without crossing a day-wide sigma threshold). `login_hour_bucket` is already on every fact record, unused by v0's baseline, so a future hour-stratified pass is a `peer-baseline.js`-only enhancement — no new collector/translator work needed to unlock it later.

### Decision 2 — peer-count DROP is deferred, and requires its own census-marker addendum (the asymmetry that makes v0 safe without one)

**A peer-count DROP toward zero would hit exactly Slice 4's must-fix-1/must-fix-2 problem, transplanted:** `factPointsFromVpnPeerEvidence` (the Slice-3 equivalent of `factPointsFromSessionEvidence`) has no marker fact analogous to `SESSION_CENSUS_MARKER_ENTITY_KEY` — confirmed by direct inspection of `fact-translators.js`'s peer section (`:303-424`): the only non-real-peer marker it ever emits is `PEER_OVERFLOW_ENTITY_KEY` (truncation only), never an unconditional per-tick "census happened, here's how many, even if zero" marker. So a genuinely-zero-active-peer tick, or a tick where one peer source degrades (e.g. `wg` becomes permission-denied, mirroring Slice 4's tmux-ok/screen-errored must-fix-2 case), produces either **no tick-group at all** (real zero, indistinguishable from "nothing happened this tick") or an **undercounted** tick-group indistinguishable from a complete one — the identical pair of hazards Slice 4 needed the Slice-1-addendum census marker to close.

**Why v0 can safely skip that addendum for a SPIKE-only detector (the load-bearing asymmetry, stated explicitly rather than left implicit):** a drop detector is vulnerable to undercount in **both directions that matter**: a partial census can *masquerade as a real drop* (false positive — exactly Slice 4's must-fix 2), and a complete drop-to-zero can be *structurally invisible* (false negative on the exact incident shape — exactly Slice 4's must-fix 1). A **spike** detector has neither vulnerability: an undercounted or entirely-absent tick-group can only ever **suppress or leave neutral** the observed count — it can never fabricate the appearance of *more* peers than were actually observed. A degraded `wg` read, a `who`/`last` failure, or a genuinely quiet zero-peer hour can only ever bias `peer.count_spike` toward **under-**alerting (a missed detection, the same class of limitation §1's "degrade-not-fabricate" already accepts everywhere else), never toward a fabricated spike. This is the one clean asymmetry that lets v0 defer the census-marker mechanism entirely without reopening either of Slice 4's must-fixes.

**Secondary, milder effect of skipping the marker (a quality/tuning concern, not a safety one, recorded honestly):** an hour with genuinely zero `observed_active` peers and no overflow marker produces no tick-group at all (mirrors the original must-fix-1 gap), so v0's windowed mean/variance is computed only over hours with at least one observed peer — this biases the baseline mean **upward** over time (true zeros never enter the window), which can only make the spike trigger **less** sensitive (a larger burst is needed to cross the same sigma threshold), never more. Recorded as a known v0 limitation, not a hidden defect.

**Decision, stated plainly: v0 = spike-only. `peer.count_drop` and the Slice-3 peer-census-marker addendum it would require are BOTH deferred to a single follow-up slice** (tentatively "Slice 4c," not scheduled), which would simultaneously (a) add a `PEER_CENSUS_MARKER_ENTITY_KEY` fact mirroring `SESSION_CENSUS_MARKER_ENTITY_KEY` exactly (unconditional per-tick marker, `census_state:"complete"|"partial"`, a Slice-3 addendum analogous to Slice 4's Slice-1 addendum) and (b) add the `peer.count_drop` rule_id plus its own must-fix-1/2-shaped exact-zero-fold and partial-skip tests — and, as a welcome side effect, would also resolve this slice's own "biased-upward baseline" quality limitation above, since the same marker unlocks real-zero-inclusion in `peer.count_spike`'s own baseline too. Recommendation: **the tighter, spike-only v0 is the right default here** — there is no incident-motivated urgency for the drop signal (the incident's peer half was a burst, not a blackout), and bundling the census-marker addendum into this slice would re-run all of Slice 4's must-fix-1/2 review surface for a signal not directly demanded by the motivating incident.

**Safety-invariant continuity, worth stating explicitly (pre-empting a plausible misreading):** Slice 3's Decision ("must-fix 2" in that slice's own numbering) resolved to "Slice 3 emits NO alert candidates in v0," pinned by two tests — (1) peer observations never write `signatures.json` (`provenance-store.js`'s store), and (2) `constraint-miner.js`'s `FAMILY_BY_FACT_NAME` allowlist (`constraint-miner.js:30-33`, today exactly `{"service.presence": ..., "network.listening_port.owner": ...}`) never gains a `peer.presence` entry. **Slice 4b does not violate either invariant.** `peer-baseline.js` is a new, deliberately-reviewed, purpose-built `extraCandidates` consumer of `peer.presence` fact-history — exactly the same relationship `session-baseline.js` already has to `session.presence` (which likewise was never added to `constraint-miner.js`'s allowlist, and never will be — `session.presence`/`peer.presence` are read directly by their own dedicated baseline modules, bypassing the generic constraint-mining path entirely). Slice 3's own Decision explicitly named this as the intended future: "alerting on unattributed/odd-hour peer logins is deferred to Slice 4 (statistical) and/or Slice 6 (correlation)" — Slice 4b **is** that promised Slice 4/statistical follow-through, not a regression of Slice 3's posture. Both of Slice 3's pinned tests remain true and unaffected; no change to `constraint-miner.js` or `provenance-store.js` is needed or made by this slice.

**Cross-reference, explicitly out of scope for this slice:** Slice 6's design (`incident-correlation.js`) already named "peer-count deviation (Slice 4b) as a third anchor type" as an explicitly deferred fast-follow (not v0) for the correlation join. Once `peer.count_spike` exists, wiring it into `incident-correlation.js` as a third anchor type (alongside `session.count_drop`/`session.churn`) is that recorded Slice-6-side follow-up — **Slice 4b itself makes no change to `incident-correlation.js`.**

### Decision 3 — fail-closed namespace (mirrors Slice 4's Decision 2 / must-fix 1 exactly)

**Not silently dropped:** `peer.count_spike` candidates are still generated, persisted to `alerts.json`, and eligible for the deterministic local delivery path (Decision 3b below) exactly like `session.*` — only S13's LLM adjudication is withheld, the same intended posture Slice 4 chose for a brand-new, unreviewed statistical family.

**The three mandatory regression tests (mirroring Slice 4's must-fix 6 exactly):**

(b) A full `adjudicateAlertNotifications` **public-seam** test (not the private `classifyAlertEligibility`): config `enabled: true`, `enabled_namespaces` set to **all** of `KNOWN_ALERT_NAMESPACES` (proving the exclusion holds under the most permissive possible opt-in, not just the one namespace `peer.*` might be mistaken for), one due `peer.count_spike` alert in the input. Assert: `status === "no_eligible_alerts"`, `excluded.unknown_namespace` counts the alert, and the injected session/LLM factory is **never invoked**.

**Explicitly FORBIDDEN (unchanged from Slice 4):** shipping an interim rule_id beginning with any currently-known consented prefix (`daemon.`, `system.`, `disk.`, `constraint.`, `provenance.`, `baseline.`, `identity.`, `correlation.`) as a stopgap.

### Decision 3b — deterministic, non-LLM local delivery (extends Slice 4's must-fix 3, not a parallel mechanism)

**The gap, restated for peers:** `peer.count_spike` is `unknown_namespace` by Decision 3 above — exactly the same "never reaches an operator except via `descartes alerts`" gap Slice 4 closed for `session.*` via `emitSessionAlertSignals`. Left unextended, this slice's first alerting family would regress to the pre-must-fix-3 state Slice 4 explicitly fixed.

**Recommended fix: extend the existing mechanism, don't build a second one.** `DETERMINISTIC_LOCAL_DELIVERY_RULE_IDS` (`session-baseline.js:48`, currently `[SESSION_COUNT_DROP_RULE_ID, SESSION_CHURN_RULE_ID]`) is already a closed rule_id **allowlist**, imported into `alert-intelligence.js` rather than restated — the natural extension is for `alert-intelligence.js` to also import `PEER_COUNT_SPIKE_RULE_ID` from the new `peer-baseline.js` and widen the allowlist to `[SESSION_COUNT_DROP_RULE_ID, SESSION_CHURN_RULE_ID, PEER_COUNT_SPIKE_RULE_ID]`. ### Decision 4 — baseline math + reuse: extract the four pure primitives, keep domain state/logic separate

**Recommendation: a scoped extraction, not a full merge and not a full duplicate.** `session-baseline.js`'s `emptyWelfordStats`, `foldWelford`, `updateEwma`, and `computeZScore` (`:192-256`) are already pure, stateless, and domain-agnostic in signature — none of them mention sessions, `ts`, or fact-history shape; they operate on a plain numeric `observedCount`. **Extract exactly these four functions (plus `DEFAULT_BASELINE_FACT_WINDOW_MS`, `session-baseline.js:70` — already generic enough that `incident-correlation.js` imports it today) into a new, tiny `tools/descartes-cli/src/welford-stats.js`.** `session-baseline.js` re-exports all five from there (`export { emptyWelfordStats, foldWelford, updateEwma, computeZScore, DEFAULT_BASELINE_FACT_WINDOW_MS } from "./welford-stats.js";`) so **zero existing import breaks** — confirmed by grep: `session-baseline.test.js:11-33` imports all four functions directly by name from `session-baseline.js`, and `incident-correlation.js:38`/`incident-correlation.test.js:11` import `DEFAULT_BASELINE_FACT_WINDOW_MS`/rule-id constants from `session-baseline.js`; a re-export preserves every one of these unchanged. `peer-baseline.js` imports the same five directly from `welford-stats.js` (not from `session-baseline.js`, avoiding a session→peer dependency in the wrong direction).

**One signature adjustment during extraction, noted so an implementer doesn't reintroduce session-specific coupling:** `computeZScore`'s `stddevFloor` parameter currently defaults to `DEFAULT_STDDEV_FLOOR` (a session-tuned PROVISIONAL constant) — in the shared module this default must be dropped (either made a required parameter, or defaulted to a tiny generic epsilon no domain module should actually rely on), so `session-baseline.js` and `peer-baseline.js` each pass **their own independently-named, independently-tunable** floor constant explicitly at their own call sites.

**Why extract-only-the-primitives rather than either extreme:** a **full merge** (sharing tick-grouping/disposition/store/candidate-building too) would recreate exactly the tight coupling this plan's own three-hash-domain discipline (session/peer/process each get their own hash domain, explicitly never shared, per Slice 3 must-fix 6) already decided against — sessions and peers have genuinely different marker semantics (peers have no census marker in v0, per Decision 2), different dimension-stratification open questions (per-`source_type` for peers, none for sessions), and different trigger signs (drop vs. spike). A **full duplicate** (re-typing `foldWelford`/`updateEwma`/`computeZScore` verbatim into `peer-baseline.js`) would fork already-tested, review-hardened math with no behavioral difference to justify the fork, and risks silent drift (a future Welford bugfix landing in one copy and not the other). Extracting just the four generic primitives is the narrow middle: real math is shared and stays single-sourced, while every domain-specific decision (what "one observation" means, what gets excluded, what the store schema looks like, what the candidate diagnostics contain) stays independently owned per module, exactly as `session-baseline.js` and `peer-baseline.js` need to evolve independently (e.g. if the deferred `peer.count_drop`/census-marker follow-up lands later, per Decision 2, it only touches `peer-baseline.js`, never `session-baseline.js`).

**`peer-baseline.js`'s own PROVISIONAL constants (independently named, not imported from `session-baseline.js`, even though the starting numeric values match Slice 4's as a reasonable default):** `DEFAULT_PEER_MIN_SAMPLE_COUNT = 30`, `DEFAULT_PEER_DEVIATION_SIGMA = 3`, `DEFAULT_PEER_CRITICAL_SIGMA = 5`, `DEFAULT_PEER_STDDEV_FLOOR = 0.5`, `DEFAULT_PEER_EWMA_ALPHA = 2/(DEFAULT_PEER_MIN_SAMPLE_COUNT+1)` — all flagged **PROVISIONAL** (mirroring Slice 4's must-fix 7): peer-count variance dynamics are a fresh, untuned surface, not something Slice 4's tuning experience transfers to "for free." **Low-variance severity-inflation risk, restated for peers (same shape as Slice 4's own stated risk, not assumed away):** a host with a stable 2-peer baseline (e.g. two permanently-connected WireGuard devices, near-zero variance floored at `0.5`) would hit `CRITICAL` (`z >= 5`) at just **2.5 additional peers** — a routine third device being added crosses that bar. Recorded as a known, deferred tuning concern, not a v0 blocker (`confidence_state` must still reach `established` first, so this is not a day-1 storm risk).

**Trigger (mirrors Slice 4's `session.count_drop`, sign flipped):** `peer.count_spike` fires when `confidence_state === "established"` **and** `z >= +DEVIATION_SIGMA` **and** `observed_count > mean_before` (defense-in-depth, redundant given a positive z, same posture as Slice 4's own drop-side guard). Severity escalates `"warning"`→`"critical"` at `z >= +CRITICAL_SIGMA`.

**`confidence_state` gate:** identical semantics to Slice 4 — `"provisional"` while `state.stats.count < DEFAULT_PEER_MIN_SAMPLE_COUNT`, no candidate ever emitted while provisional (day-1/cold-start no-storm).

**Windowed recompute, gradual-drift, regime-change — all inherited verbatim from Slice 4's must-fix 5 design, just re-derived over `peer.presence`:** the Welford/EWMA accumulator recomputes fresh over the complete (non-overflow) tick-groups currently inside the same `DEFAULT_BASELINE_FACT_WINDOW_MS` (~31-day) window on every fold, never an incrementally-accumulated running total — same auto-forgetting property, same self-correcting regime-change behavior, same z-score-computed-against-the-window-excluding-the-tick-under-test ordering discipline. Overflow-tick handling reuses the already-shipped `PEER_OVERFLOW_ENTITY_KEY` marker (`fact-translators.js:326`) exactly like Slice 4 reused `SESSION_OVERFLOW_ENTITY_KEY`: an overflow tick-group is excluded from the recompute (its true count is only bucketed, never exact), `last_folded_ts` still advances past it, `skipped_overflow_tick_count` increments.

## Slice 5 — Real-time event-source spike (TIME-BOXED, decision-aid only)

**Completed: 2026-07-14 — GO/NO-GO: DEFER.** Full decision document: `docs/plans/2026-07-14-event-source-spike.md`. Summary: neither `eslogger` (macOS) nor passive-auditd-read (Linux) closes the incident's actual gap without new privilege — `eslogger` needs no new *entitlement* (it's an Apple-signed binary that already carries the ES entitlement) but does need a standing root+Full-Disk-Access grant, and its event coverage is strong for exec+login but structurally **zero** for IP-network events (confirmed both by direct read-only probing on the dev machine — `eslogger --list-events` lists only `uipc_bind`/`uipc_connect`, Unix-domain-socket-only — and by research: this is zero achievable via any **supported** EndpointSecurity path, not merely unbuilt, so no custom ES client closes it either — though a corroborated-but-undocumented macOS 26.4 rumor of new reserved ES network events is an explicit revisit trigger, see the decision doc §1b/§5). Linux passive auditd read is real (`CAP_AUDIT_READ`, narrower than `CAP_AUDIT_CONTROL`/root) but contingent on the operator's own pre-existing, Descartes-uncontrolled audit-rule configuration (upstream `auditd.conf` defaults `log_group=root`, so a group-readable log-tail path is an additional opportunistic route, not a default one). Linux eBPF is DECLINEd as disproportionate to this milestone (its own S3-priv-scale privileged build); Linux fanotify is DECLINEd as a **weak exec signal at high privilege cost and zero network** — it *does* have an exec event type (`FAN_OPEN_EXEC`/`FAN_OPEN_EXEC_PERM`, since Linux 5.0, used in production by RHEL's `fapolicyd`), but with no argv/process-ancestry context and blind to interpreter/script reads, dominated by auditd/eBPF at similar-or-lower privilege (corrected 2026-07-14, Fable review — the original "no exec event type at any privilege level" claim was factually wrong, drawn from `fanotify_init(2)` alone rather than `fanotify_mark(2)`/`fanotify(7)`). Per-branch: DECLINE macOS custom-ES-client and all network-event coverage (revisit if Apple documents macOS 26.4's ES network events as real, published API); DEFER macOS `eslogger` (exec+login only), now **trigger-gated** — actionable only on a real incident where Slices 1–4/6 demonstrably failed to reconstruct what mattered, or an operator explicitly requesting sub-tick forensics and accepting the standing root+FDA grant — and, once triggered, a bounded/alert-triggered redesign (extending Slice 2's evidence-freeze model) plus an operator-run manual validation capture, first; DEFER Linux auditd passive-read as a low-priority opportunistic future collector. **No Slice 5b build is scheduled.** Zero daemon-loop code shipped from this slice, per its own Definition of Done. **Fable-reviewed 2026-07-14: ENDORSE-WITH-CHANGES** — 4 factual/framing corrections folded in (fanotify, macOS network claim + revisit trigger, eslogger DEFER trigger-gating, auditd group-readable resolution); no branch verdict flipped; full corrections in the decision doc.

**locallyTestable:** the macOS half is investigable on this dev machine; the Linux half is NOT (no local VM) — constrained to documentation/community-report research plus, if warranted, a disposable Linux CI probe step mirroring the already-proven `scripts/probe-linux-ci-capability.sh` precedent. This asymmetry should tighten the Linux half's time-box further, not loosen the macOS half's.

**This is explicitly a spike, per the project's spike guardrail: a decision aid ending in a written go/no-go, not a battle-hardened prototype. "Skip it, gap too narrow" is a valid, successful outcome** — exactly the posture the S3-priv macOS spike already modeled in this repo (`docs/plans/2026-07-11-s3-priv-elevated-read-path.md` §"macOS go/no-go spike").

**Goal:** determine whether Descartes should invest in a real **event stream** (macOS EndpointSecurity/`eslogger`; Linux auditd/eBPF/fanotify) to reconstruct a succession of events after the fact — every collector in Slices 0–4 is point-in-time/polling and cannot answer "what exactly happened between tick T and T+1hr," which is precisely the retrospective-reconstruction gap the incident review named as unclosed.

**Doors-and-corners flag (mandatory before any follow-on build, not just at the end):**
- **macOS EndpointSecurity** requires a system extension, an Apple-approved `com.apple.developer.endpoint-security.client` entitlement (not self-grantable), and notarization implications for a background daemon running in an elevated launch context — a large lift. **`eslogger`** (Apple's own EndpointSecurity-backed CLI, ships with macOS, readable without a custom entitlement if the invoking process/user has appropriate access) may be a far lower-lift substitute worth spiking *first*, before considering a custom ES client at all.
- **Linux auditd** is a system-level configuration Descartes would either need to *configure* (a mutating action — explicitly out of scope for a read-only agent) or passively *consume* if the operator has already configured it (read-only, acceptable). **Nice-to-have (degrade case for the spike doc):** even the read-only `auditctl -l` (listing existing audit rules) typically requires `CAP_AUDIT_CONTROL`/root on most distributions — passive consumption is not automatically unprivileged just because it's read-only; the spike's decision document should record this as an expected degrade case (non-root operator sees `unable`/permission-denied) rather than assuming passive auditd read is always available. **eBPF/fanotify** require `CAP_BPF`/`CAP_SYS_ADMIN`-class privilege — this entire branch of the spike lives in "privileged elevated path" territory akin to S3-priv, and if the spike concludes "go," the resulting build must reuse that plan's opt-in/audience-scoped/deny-by-default model, not invent a new privilege mechanism.

**Time-box:** propose a strict bound (e.g. 1–2 investigation sessions). Output: a written go/no-go document (mirroring the S3-priv spike doc's own structure — a handful of real captures + a decision, not a hardened POC). Decision criteria to record: (a) does `eslogger`/passive-auditd-read already cover the incident's actual gap (exec + network + login events) without new entitlements/privilege; (b) operational cost (log volume, always-on daemon, CPU/battery) versus the marginal forensic value over Slices 1–4's polling approach; (c) an explicit recommendation — proceed to a real build slice, defer, or decline.

**Safety checks for the spike itself:** read-only investigation only — use already-installed OS tools (`eslogger`, `auditctl -l` in read mode) to observe; do **not** configure new audit rules or install a system extension during the spike — that is itself an install-time decision requiring its own review, at the same bar as S3-priv's `root_helper`. No code from this slice runs by default. A fresh `doors-and-corners` pass is required before any follow-on build slice even begins design, not merely before it ships.

**Definition of done:** a written go/no-go decision document under `docs/plans/` or `docs/solutions/`, explicitly time-boxed, zero daemon-loop code shipped from this slice.

**Platform coverage:** macOS and Linux investigated in parallel; the Linux half is documentation/CI-probe-only given the dev-machine constraint.

## Slice 6 — L2 incident-correlation (reuses the S13 gated LLM path — no new externalization surface)

**Deepened 2026-07-14 (design pass; Fable-reviewed 2026-07-14, 7 must-fixes folded in — see the dated note directly below and "Flagged for review" at the end of this section):** ground-mapped against the shipped S13 gate and the shipped Slice 1/3/4 fact/alert shapes (verified by direct code reads, cited by file:line throughout). v0 ships **exactly one** deterministic correlation pattern — `correlation.login_kill_proximity` — joining Slice 4's `session.count_drop`/`session.churn` alert-history against Slice 3's `peer.presence` fact-history within a bounded window. A **new, real, consentable, default-off** `correlation` namespace is added to the S13 gate (unlike Slice 4's `session.*`, which is permanently `unknown_namespace` and can never be consented at all) — this is the one place in this whole milestone that deliberately opens a new LLM-eligible surface, so Decision 2 below is the load-bearing decision of this slice. A load-bearing grounding finding, not previously recorded anywhere in this plan: Slice 3's peer-identity `provisional → known_good` state machine (`peer-signature-store.js`'s `reconcilePeerSignatures`/`applyPeerIdentityObservation`) is **deliberately left unwired into the daemon loop** — confirmed by that module's own SCOPE NOTE ("nothing in `daemon.js` calls `reconcilePeerSignatures`/`writePeerSignatureStore`... wiring an actual reconcile-on-tick call... is left for Slice 4/6"). Slice 6 v0 does **not** pick up that wiring (it would be a second, separately-reviewable piece of new persisted-state machinery bolted onto a slice that is supposed to stay a pure fact/alert-history join) — instead, "unattributed-looking" is approximated by a cheap, explicitly-provisional novelty count computed inline, purely from already-persisted `peer.presence` fact-history. This is recorded as a conscious, documented scope decision (Decision 1), not a gap discovered later.

**Reviewed (Fable) 2026-07-14:** 7 must-fixes folded in.

**locallyTestable:** true.

**Goal:** cross-stream timeline join — correlate Slice 4's session-drop/churn alert-history against Slice 3's peer-login fact-history within a bounded time window (the incident's own shape: "login-proximity ∧ kill-proximity") — producing a ranked hypothesis. This is **mostly deterministic L0/L1 correlation logic**; the LLM is used **only** as the final gated adjudicator over an already-assembled, already-sanitized candidate, reusing `alert-intelligence.js`'s S13 machinery verbatim.

**Does not depend on Slice 5.** The correlation join operates over already-persisted fact-history/alert-history deltas (a session disappearing between two ticks is itself a fact-history event; no continuous event stream is required to detect that it happened, only to pinpoint exactly when within the tick window). Slice 5's outcome would enrich a future correlation pass's time resolution, not gate this one.

**Correlation-join data source constraint (folded in via Slice 4's 2026-07-14 Fable review, plan-text addition iii — restated and made concrete below):** the correlation join operates strictly over persisted **fact-history** (`fact-store.js`'s `readFactPoints`) and **alert-history** (`alert-store.js`'s `readAlertRecords` — every row in `alerts.json`, of **any** `status`: `active`, `recovered`, `acknowledged`, or `suppressed`) — **never** over the live in-memory candidate array a given daemon tick is in the middle of computing. Concretely: `computeCorrelationCandidates` calls `readAlertRecords`/`readFactPoints` itself (the same way `computeProvenanceIdentityCandidates` reads its own store independently of `evaluateAndPersistAlerts`'s internal `existing`-alerts fetch) — it is never handed "this tick's active alerts" as an argument, and it does not require `session.count_drop`/`session.churn`'s own `extraCandidates` entry to have run first in the same tick. A kill-side alert row with `status:"recovered"` (the session count already bounced back, or a churn pair aged out) is just as valid a correlation anchor as one still `status:"active"` — correlation is about whether the two events **happened** near each other in time, not about whether the kill-side alert still happens to be flagged live right now.

### Decision 1 — what is correlated + the deterministic join

**v0 ships exactly one pattern, `correlation.login_kill_proximity`, and nothing else.** Two independently-sourced anchors, joined on time:

- **Kill-side anchor:** every row from `readAlertRecords(descartesPaths)` whose `rule_id` is `session.count_drop` or `session.churn` (Slice 4's `SESSION_COUNT_DROP_RULE_ID`/`SESSION_CHURN_RULE_ID`) **and** whose `first_seen` falls within `DEFAULT_CORRELATION_LOOKBACK_MS` (provisional default **24h**) of `now`. `first_seen` is the tick at which `applyAlertCandidates` first created that alert id (`alert-store.js:230-235`) — an honest proxy for "when the deviation was first detected," off by at most one structural-tick cadence (~1h), which is already inside the noise floor of every stream this slice reads. The lookback bound exists for two reasons: (a) cost — `alerts.json` has no retention/pruning of its own (a pre-existing characteristic, not introduced by this slice — flagged as a new observation for a future open question, not solved here), so an unbounded scan would grow with the file's entire lifetime; (b) correctness — an ancient, long-recovered deviation should not resurrect as a "new" correlation months later just because a peer login happens to be discovered near it in a batch replay.
- **Login-side candidates:** `peer.presence` fact points from `readFactPoints(descartesPaths, { now, windowMs: DEFAULT_BASELINE_FACT_WINDOW_MS })` (reusing Slice 4's own 31-day constant for consistency — one read covers both the proximity join and the novelty count below, no second I/O class). For each kill-side anchor, a peer observation **qualifies** iff **all** of:
  1. **Proximity:** `abs(peerTickTs - anchor.first_seen) <= DEFAULT_CORRELATION_WINDOW_MS` (provisional default **3h**, reusing `peer-signature-store.js`'s own already-established `DEFAULT_PEER_PRESENCE_WINDOW_MS` value for consistency rather than inventing a new arbitrary number).
  2. **Odd-hour:** the peer fact point's `login_hour_bucket` attribute (Slice 3's closed-enum "00".."23" or "unknown", `fact-translators.js:345-354`) is in `CORRELATION_ODD_HOURS` (provisional default `{"00","01","02","03","04","05"}`, local time). **Corrected 2026-07-14 (Fable review, MUST-FIX 6) — semantics restated precisely:** despite its name, this attribute is **the hour of the observation TICK**, not the hour of the peer's actual login instant — `fact-translators.js:345-349`'s own doc comment states it is "the hour at which THIS observation tick occurred... Deliberately NOT derived from parsing an exact login instant." Earlier drafts of this plan described it as "the peer LOGIN hour," which would mislead both the LLM and the operator into believing this pins a login timestamp it does not. The correlation candidate's own diagnostic for this value is therefore named `peer_observed_hour_bucket`, not `peer_login_hour_bucket` (see the Diagnostics bullet below). `"unknown"` is a possible value (a peer point whose `ts` fails to parse) and, being outside `CORRELATION_ODD_HOURS`'s closed set of hour-strings, never qualifies as odd-hour — stated explicitly rather than left as an implicit consequence of set non-membership.
  3. **Unattributed-looking (the novelty proxy, NOT full attribution — see the header note above), GATED against cold-start inflation (Fable review, MUST-FIX 4, hard requirement):** counting only tick timestamps strictly before the qualifying observation's own `ts`, within the same already-read fact-history window, this peer `entity_key` appears at **`<= CORRELATION_NOVELTY_MAX_PRIOR_TICKS`** (provisional default **1**) distinct prior ticks — **but only once this peer's own fact-stream has enough history for "few prior ticks" to be a meaningful signal rather than a week-1 deployment artifact.** Require a minimum peer-fact-stream depth before ANY peer may qualify as "unattributed-looking" at all: at least `CORRELATION_MIN_PEER_HISTORY_TICK_GROUPS` (provisional, implementer-set, e.g. 4) distinct tick-groups for that peer's `entity_key`, spanning at least `CORRELATION_MIN_PEER_HISTORY_DAYS` (provisional, implementer-set, e.g. 3 days) within the read window; if a peer (or the window as a whole) doesn't meet this bar, that peer emits no correlation candidate at all — fail closed to "no signal," never fail open to "assume novel." **Why this is a hard requirement:** in the week after this slice first deploys, every peer has at most 0-1 prior ticks purely because the fact-history itself is young — under the un-gated rule, EVERY peer would look "novel/unattributed" during week 1, producing false correlations exactly when the operator has the least history to judge them by. A brand-new-to-this-window peer (0 prior ticks) or one seen exactly once before (1 prior tick) counts as "unattributed-looking" **only once the minimum-history bar above is independently satisfied**; a peer with a long, repeated presence history does not qualify either way. **This remains explicitly weaker than real attribution** (it says "rarely seen recently, and we have enough history to say so," not "never confirmed via a multi-iteration `provisional → known_good` promotion") — stated as a conscious v0 simplification, not an oversight.
  4. **Not overflow-degraded (Fable review, MUST-FIX 5, hard requirement):** if the fact-history window used for the novelty count contains any `peer.presence` tick whose `entity_key === PEER_OVERFLOW_ENTITY_KEY` (`"peer.overflow-marker.v1"`, `fact-translators.js:323` — currently module-private; exported as part of this slice's own `fact-translators.js` change, see Decision 6) — i.e. the peer-census collector hit its own cap on at least one tick within the window — the prior-tick count for every peer in that window is a potential UNDERCOUNT (an unknown number of real observations were folded into the overflow marker instead of being recorded per-entity), so no peer may qualify as "unattributed-looking" from an overflow-degraded window; `findQualifyingPeerObservations` returns no qualifying observations for any anchor whose read window contains an overflow tick. This mirrors Slice 4's own overflow fold-skip discipline exactly (`session-baseline.js`'s tick-group `hasOverflow` check, which excludes overflow-degraded tick-groups from statistical recompute rather than silently under-counting into the baseline).
- **Ranking + candidate cap (no-storm discipline, Decision 5):** for a given anchor, **at most one** correlation candidate is ever emitted, even if several peer observations qualify — the qualifying observation with the smallest `abs(delta)` wins (ties broken by lexicographically-smaller `entity_key`, for determinism); the count of all qualifying observations for that anchor is preserved as a diagnostic (`candidate_pool_size`), giving the adjudicator a cheap signal of hypothesis ambiguity without needing a second candidate per peer. This bounds worst-case emission to "at most one new alert per kill-side anchor," never a combinatorial peer × anchor explosion.
- **Fingerprint/id:** `fingerprint = `${anchor.fingerprint}__${peerEntityKey}`` (both components already bounded/safe strings per Slice 4's/Slice 3's own construction — `anchor.fingerprint` is `"global"` for `session.count_drop` or an entity-key hash for `session.churn`; `peerEntityKey` is already `peer.<source_type>.<16hex>`); `id = alertId(CORRELATION_RULE_ID, fingerprint)` (`alert-store.js:27-30`, unchanged, reused as-is).
- **Diagnostics (all through `sanitizeDiagnostics`, Decision 5):** `kill_rule_id` (closed enum, one of the two Slice 4 rule_ids), `anchor_fingerprint` (already a hash/enum string), `peer_entity_key` (already a hash string), `peer_source_type` (closed enum `wireguard|ssh|vpn_service|unknown`), `peer_observed_hour_bucket` (closed enum "00".."23" or "unknown" — **renamed from `peer_login_hour_bucket` 2026-07-14, Fable review MUST-FIX 6**: this is the observation tick's hour, not the peer's login instant; see the "Odd-hour" bullet above), `proximity_seconds` (finite number — a **duration**, not an absolute timestamp, so it carries none of the occupancy-signal sensitivity a raw clock time would; mirrors Slice 4's own `z_score`/`mean_before` numeric diagnostics — worth noting explicitly, per Fable review nice-to-have (ii), that `compactAlert` already ships `first_seen`/`last_seen` as raw ISO timestamps for every namespace's own alert record, so absolute clock times are not somehow absent from the prompt; the point here is narrower — this field specifically doesn't need to duplicate that channel with a second, correlation-specific absolute timestamp, since a bounded duration already answers the one question this diagnostic needs to answer), `peer_novelty_prior_tick_count` (finite integer), `candidate_pool_size` (finite integer), `anchor_severity` (closed enum `info|warning|critical` — the anchor's own original stored severity, preserved here so the LLM/operator retain that context even though the correlation candidate's own stored severity is capped at `"warning"` in v0, see the Severity bullet below, Fable review MUST-FIX 7).
- **Severity — CAPPED at `"warning"` in v0 (Fable review, MUST-FIX 7, hard requirement; resolves this section's own "Flagged for review" item 1 severity sub-question — see the updated item below):** the correlation candidate's **stored** severity is capped at `"warning"` unconditionally in v0 — it is never automatically inherited as `"critical"`, even when the kill-side anchor is itself a critical `session.count_drop`. (Previously this plan inherited severity directly from the anchor with no cap — corrected.) **Why the cap:** naive inheritance would import Slice 4's own admitted-PROVISIONAL Welford/EWMA sigma constants' inflation risk directly into the shared alert budget's CRITICAL lane — a false-critical `session.count_drop` (a plausible outcome of an untuned provisional constant) would mint a critical correlation candidate that preempts notification ordering and consumes the shared hourly critical-reservation slot (`DEFAULT_ALERT_INTELLIGENCE_CRITICAL_RESERVATION`, `alert-intelligence.js:15`) meant for a genuine emergency. Capping the correlation family's own stored severity at `"warning"` removes that inflation path structurally. The anchor's real severity is not lost — it is preserved as the `anchor_severity` diagnostic field (Diagnostics bullet above) for LLM/operator context. **The LLM's own `decision.severity` (once consented) stays free, unaffected by this cap** — exactly the existing `alert_severity`-not-`decision.severity` discipline already in place for every namespace (`alert-intelligence.js:198-206`): the model can still describe a finding as urgent in its own adjudication text; only the alert's *stored*/budget-classified severity is capped. **This cap lifts once Slice 4's sigma constants move from PROVISIONAL to tuned-against-real-data** (tracked as a follow-on, not designed here).
- **Re-emission / recovery, no new store:** `computeCorrelationCandidates` is a **pure function over already-persisted history, with no state file of its own** — exactly like `detectSessionChurn`. It re-derives the same candidate (same `id`) every tick for as long as its anchor keeps satisfying the lookback bound; once the anchor's `first_seen` ages out of `DEFAULT_CORRELATION_LOOKBACK_MS`, the candidate is simply absent from the next tick's array and `applyAlertCandidates` marks it `"recovered"` automatically — the existing recovery machinery, unchanged, does all the work.
- **Explicitly out of v0 scope (deferred, not designed here):** correlating against `provenance.*`/`identity.*` alert-history (deleted-exe/public-bind/unknown-identity/identity-drift warnings) as a second correlation pattern; peer-count deviation (Slice 4b) as a third anchor type. Recorded as fast-follow candidates, mirroring Slice 4's own "bank the simplest pattern first" sequencing logic (§3) — v0 stays to the exact incident shape this whole milestone is motivated by.

```js
if (id.startsWith("correlation.")) return { namespace: "correlation", hardExcluded: false };

The plan's earlier "zero `alerts.js` edits" claim is corrected to: zero validation-logic edits, one disclosure-data-map entry required. **Hard requirement, restated from the task framing:** the operator must explicitly run `descartes alerts intelligence enable-namespace correlation` before a single correlation candidate is ever adjudicated by the LLM — this is not a soft default, it is the same structural gate every other learned namespace already goes through, unmodified.

```

### Decision 3 — prompt-boundary safety (defense-in-depth re-sanitization + the sole-call-site test)

**Defense-in-depth `sanitizeDiagnostics()` re-run at `compactAlert` (must-fix 3, hard requirement, module-wide fix — corrected 2026-07-14 per Fable review MUST-FIX 1, see below):** `compactAlert` (`alert-intelligence.js:332-344`) currently passes `alert.diagnostics` **verbatim** into `diagnostics: alert.diagnostics` — confirmed by direct read, there is no re-validation between what a candidate builder wrote to `alert-store.js` and what `alertIntelligencePrompt`'s `JSON.stringify` embeds into the prompt text. Change:
```js
diagnostics: sanitizeDiagnostics(alert.diagnostics),
```
requiring a new `import { sanitizeDiagnostics } from "./diagnostics-sanitizer.js";` at the top of `alert-intelligence.js` (not currently imported there — confirmed by reading the file's import block). **This fix is module-wide, not correlation-scoped** — every namespace's `compactAlert` output gets re-sanitized, not just `correlation`'s. This is deliberate: Slice 6 is the first slice whose candidate diagnostics are assembled from **two** independently-hashed upstream streams (Slice 3's peer facts, Slice 4's session-baseline alert), which is exactly the kind of composition point where a subtle upstream hash-at-source gap (a future translator bug) would first become externally visible, so the backstop belongs at the shared choke point (`compactAlert`), not duplicated per-namespace.

**CORRECTED 2026-07-14 (Fable review, MUST-FIX 1) — this is NOT a no-op for the `metric` namespace:** this plan previously claimed every existing candidate builder already routes its diagnostics through `sanitizeDiagnostics` at write time, making the re-run idempotent-by-construction. That claim is **false** for the `metric` family: `alert-store.js`'s `evaluateAlertRules` (`alert-store.js:57-159`) never calls `sanitizeDiagnostics`, and `alert-store.js` does not even import `diagnostics-sanitizer.js` (confirmed by reading its import block: only `node:crypto`/`node:fs`/`node:path`/`history-store.js`). So this change IS a real prompt-content change for the DEFAULT-ENABLED `metric` namespace, in two ways — both fail SAFE (over-redaction, never a leak), which is why the fix is still correct to ship, but the plan must say so plainly rather than assume idempotency:
  1. **`undefined` values become redaction markers.** `daemon.samples.missing`'s candidate (`alert-store.js:96-101`) can emit `diagnostics: { window_ms: undefined, point_count: ... }` when `historySummary` itself is absent — `JSON.stringify` silently drops an `undefined`-valued key today (so the pre-fix prompt simply omits `window_ms`), but post-fix, `sanitizeDiagnostics`'s `classifyValue` classifies `undefined` as `unsupported_type` and emits a `{redacted:true, reason:"unsupported_type", original_length:undefined}` marker in its place — a new key appears in the prompt where none did before.
  2. **`sanitizeDiagnostics` is NOT idempotent on already-redacted content.** `normalizeAlertRecord` (`alert-store.js:182`) admits any object-shaped `diagnostics` value verbatim, including a nested object — so a diagnostics value that is *already* a write-time redaction marker (e.g. `{redacted:true, reason:"unsafe_string_shape", original_length:12}`) is not itself a number/boolean/safe-string/hex-hash, so `compactAlert`'s re-run reclassifies it as `unsupported_type` and **rewrites** it to `{redacted:true, reason:"unsupported_type", original_length:undefined}` — the original redaction reason and length are lost. Re-running `sanitizeDiagnostics` on its own output is not a fixed point today.

**RECOMMENDED FIX (Fable review, MUST-FIX 1(ii)):** teach `sanitizeDiagnostics` (`diagnostics-sanitizer.js`) to recognize a WELL-FORMED redaction marker shape (an object with exactly `{redacted:true, reason:<safe enum string>, original_length:<finite number or undefined>}`) and pass it through unchanged rather than reclassifying it — restoring true idempotency (a redacted value re-sanitizes to the identical marker, not a re-redacted one) instead of merely arguing it from `classifyValue`'s general logic. This is a small, additive change to `diagnostics-sanitizer.js` (a marker-shape check ahead of the generic `classifyValue` dispatch), not a new file — see Decision 6's "Files to change."

**Fixtures required (Fable review, MUST-FIX 1(iii)), replacing the single "all-safe numbers" idempotency fixture as insufficient:**

### Decision 4 — determinism vs LLM split (confirming un-consented behavior against S13's existing design)

**The join, candidate, and ranking are 100% deterministic and happen regardless of consent.** `computeCorrelationCandidates` has no knowledge of `alert-intelligence.json`'s `enabled_namespaces` at all — it is wired into `daemon.js`'s `extraCandidates` exactly like every sibling source, gated only by the shared `learned.json` kill switch (Decision 5), and always produces (or doesn't produce) a `correlation.login_kill_proximity` candidate purely from the deterministic rule in Decision 1. The LLM, reached only through S13's already-shipped `adjudicateAlertNotifications` → `createPrivateAlertSession(..., {enableTools:false})` path, is invoked **only** to decide `notify`/`severity`/`title`/`body`/`reason` over an already-fully-formed hypothesis — it never gathers additional evidence (`enableTools:false` holds, pinned by Decision 3's sole-call-site test) and never runs unless `correlation` is both consented **and** within that hour's shared budget.

**When NOT consented — confirmed to match S13's existing behavior for every other known-but-unconsented namespace, by direct code read:** because `correlation` is a **known, registered** namespace (unlike Slice 4's permanently-`unknown_namespace` `session.*`), an un-consented correlation candidate's `classifyAlertEligibility` reason is `"not_consented"` (`alert-intelligence.js:489-497`) — the exact same code path an operator hits today by simply never running `enable-namespace baseline`/`provenance`/`identity`. The candidate is still generated every tick, still persisted to `alerts.json` via the normal `evaluateAndPersistAlerts`/`applyAlertCandidates` flow, and still visible via `descartes alerts` — it is simply excluded from `adjudicateAlertNotifications`'s eligible set before any LLM/budget accounting happens (`:563-568`), identically to how a `baseline.session_count_drop`-shaped candidate would behave if `baseline` were disabled.

**Considered and explicitly rejected: a Slice-4-style deterministic local-delivery bypass for `correlation`.** Slice 4 needed `emitSessionAlertSignals`/`DETERMINISTIC_LOCAL_DELIVERY_RULE_IDS` (`alert-intelligence.js:283-330`) specifically because `session.*` is **permanently** `unknown_namespace` — there is no consent mechanism that could ever surface it via the normal LLM/notification path, so without a deterministic bypass the milestone's first alerting slice could never actively notify anyone. `correlation` has no equivalent gap: it is a real, opt-in-able namespace, and an operator who wants active notification for it has exactly one well-worn lever — `descartes alerts intelligence enable-namespace correlation` — after which the existing S13 delivery path (adjudicate → `deliverNotificationDecision` when `decision.notify`) already handles it, no new code required. Building a second bypass here would blur the very distinction Decision 4 is supposed to keep sharp (deterministic-and-always-local vs. consentable-and-LLM-adjudicated) — rejected.

**Critical-budget-reservation invariant — now structurally protected against inflation by this family in v0 (updated 2026-07-14, Fable review MUST-FIX 7):** `correlation` candidates compete for the same shared hourly `max_calls_per_hour`/`critical_reservation` budget as every other namespace, but because the correlation family's own stored severity is capped at `"warning"` (Decision 1), a correlation candidate can never itself consume the dedicated `critical_reservation` slot in v0 — that reservation stays available for a genuine critical `metric`/`baseline`/`identity`/`provenance` finding. Whether correlation should get its own separate reservation carve-out once the severity cap eventually lifts remains open (§5 Open Question 9, left open, not decided in this pass) — no new reservation carve-out ships in v0.

### Decision 5 — no new leak / no storm

- **Gated by `learned.json` before any I/O**, identical short-circuit shape to every sibling: `const learnedConfig = await loadLearnedConfig(descartesPaths); if (!learnedConfig.enabled) return [];`, checked before `readFactPoints`/`readAlertRecords` are ever called.
- **Cannot storm on historical data at first run.** Unlike Slice 4's must-fix 4 (which had to add an explicit recency bound to stop weeks of pre-existing fingerprint-change history from replaying as a churn storm on upgrade day), Slice 6's own `DEFAULT_CORRELATION_LOOKBACK_MS` (24h) bound means the very first tick after this slice ships can derive **at most one candidate per kill-side anchor already within the last 24h** — bounded, small, and arguably a genuinely-wanted immediate surfacing of a real recent coincidence, not a storm replaying weeks of history. **Still requires its own pinned fixture** (hard requirement, not merely argued): a fact/alert-history fixture containing several weeks of old, otherwise-qualifying anchor+peer pairs entirely outside the 24h lookback asserts **zero** candidates fire on the first call — pinning the boundedness rather than resting on the argument alone.
- **Day-1 no-storm (ordinary case):** with no session/peer history at all (a fresh `learned.json`-enabled install), `readAlertRecords`/`readFactPoints` both return empty, and `computeCorrelationCandidates` returns `[]` — no candidate, no throw.
- **Bounded-window discipline is itself tested**, not assumed: a fixture with the two events far outside `DEFAULT_CORRELATION_WINDOW_MS` (3h) asserts **no** spurious correlation.
- **Candidate diagnostics are numeric/hash/closed-enum only**, enforced twice: once by the candidate builder's own `sanitizeDiagnostics()` call (matching every sibling family) and again, defense-in-depth, by Decision 3's `compactAlert` re-run — belt and suspenders across the one composition point in this milestone that assembles diagnostics from two independently-hashed streams. (Decision 3's re-run is corrected 2026-07-14, Fable review MUST-FIX 1, to also pass a well-formed redaction marker through unchanged — see Decision 3 for the fixtures this requires.)
- **Overflow-degraded windows never qualify for the novelty proxy** (Decision 1, Fable review MUST-FIX 5) — a fixture-pinned fold-skip, not merely an argued property.
- **Read-cost note, not a new class of cost:** `computeCorrelationCandidates` performs its own `readAlertRecords`/`readFactPoints` calls (independent of `evaluateAndPersistAlerts`'s own later re-read of `alerts.json` for the `existing`/merge step) — this mirrors every other `extraCandidates` source's existing independent-read pattern (e.g. `computeProvenanceIdentityCandidates` already reads `signatures.json` on its own), so it is not a new architectural cost class, only one more small-file read per tick. `alerts.json` having no retention/pruning mechanism of its own is a **pre-existing** characteristic (confirmed: no retention constant anywhere in `alert-store.js`, unlike `fact-store.js`'s 30-day/5MB cap) that this slice's own lookback bound (24h) already protects it from — but its unbounded on-disk growth over the system's lifetime is a separate, pre-existing question this slice surfaces rather than solves; recorded as a new addendum to §5's open questions, not fixed here.

## 5. Open Questions (operator input needed)

1. **Fleet/multi-host topology.** Descartes is single-host by design. Does "VPN peer" in Slice 3 stay scoped to peers observed *from this host's* VPN interface, or is there a latent multi-host correlation need (e.g. correlating logins across several monitored Descartes instances)? Recommendation: stay single-host; flag if that's wrong.
2. **`collect_recent_logs` new ceiling (Slice 0). RESOLVED 2026-07-13:** 72h (`MAX_WINDOW_MINUTES=4320`), shipped; single source of truth via the pi-harness schema importing the constant (not made configurable this pass). Remaining sub-question → **Slice 0b (deferred):** a genuinely useful 7-day window needs pre-computed aggregates, not raw sampling under the 200-event cap.
3. **Authority-store confirmation identities (Slice 7).** The hardest unresolved question in this whole plan: where do "multi-party confirmation" identities actually come from for a future containment action? Multi-device push, a second human operator, or a time-delay proxy? This blocks any real Slice 7 design, not just its implementation.
4. **Session-census sampling cadence (Slice 1).** Hourly structural tick (v0 default, zero new plumbing) vs. fast tick (better time-resolution for pinpointing a mass-drop, requires new daemon plumbing). **Sharpened per must-fix 6:** a count-only hourly census is not merely coarse but structurally **invisible** to a same-hour kill-then-resurrect — the actual shape of the motivating incident. Slice 1's bucketed `created_at` attribute (must-fix 6) gives the hourly cadence a way to detect that *some* churn happened within the tick even without a faster sampling rate, but it does not restore minute-level timing precision. Worth revisiting whether bucketed-attribute churn detection is sufficient or whether the fast-tick plumbing is still needed, once real hourly-resolution data is examined against a real (or simulated) incident timeline.
5. **Session-count baseline namespace scoping (Slice 4).** **No longer a soft recommendation — must-fix 1 makes closing the consent-reuse leak a hard requirement of Slice 4 itself** (a session-count deviation candidate must not silently ride an operator's existing `baseline` opt-in). What remains genuinely open is *which* of must-fix 1's two acceptable designs to ship: the v0-default fail-closed `session.` rule_id prefix (simpler, ships a safe local alert immediately, no LLM adjudication until a deliberate namespace exists) versus the full `baseline.session.` classifier-branch-plus-registration alternative (more machinery, enables LLM adjudication sooner but requires the longest-prefix-first ordering fix). This plan defaults to the v0 fail-closed design as the safer starting point but leaves the timing of the full-machinery alternative as an operator call.
6. **VPN technology scope (Slice 3).** Which VPN technologies does the operator actually run (WireGuard? something else? none)? The collector's real-world coverage should be driven by this rather than attempting universal support speculatively.
7. **Real-time event-source priority (Slice 5).** Is the spike scheduled now, or deferred until Slices 1–4/6 are further along and the actual residual retrospective-reconstruction gap can be assessed against real usage?
8. **Evidence-bundle retention (Slice 2).** Should evidence-freeze bundles ever expire/roll off like fact-history's existing 30-day/5MB cap, or are they exempt as potential legal-hold material? This plan defaults to "never auto-delete" but flags it as a decision, not a settled policy.
9. **Correlation-candidate LLM budget (Slice 6).** Should `correlation` namespace candidates compete for the shared hourly S13 budget, or warrant their own reservation (mirroring the existing critical-severity carve-out), given a correlation finding is plausibly higher-value per call than a routine metric deviation?
10. **`alerts.json` retention (surfaced during Slice 6 deepening, not introduced by it).** Confirmed by direct read of `alert-store.js`: unlike `fact-store.js`'s 30-day/5MB retention cap, `alerts.json` has no retention/pruning mechanism of its own — recovered/resolved rows persist indefinitely (only their `status` field changes). This was already true before Slice 6; Slice 6 merely adds a second reader (`computeCorrelationCandidates`'s own `readAlertRecords` call, bounded on its own read by `DEFAULT_CORRELATION_LOOKBACK_MS`) and makes the gap newly visible. Not solved in this plan — flagged for a future pass, same disposition as Open Question 8's evidence-bundle retention question.
